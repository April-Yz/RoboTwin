# V1.14 Preview Top1 Execution Relation

This note records the current execution behavior for the preview-summary-driven AnyGrasp planner path.

Related code:
- `code_painting/plan_anygrasp_keyframes_r1.py`
- `code_painting/plan_anygrasp_keyframes_r1_batch.py`

## 1. Selection Source

In this mode, the planner does not rescan all AnyGrasp candidates during keyframe selection.

Instead it reads preview output JSON directly:
- `summary.json` from `render_anygrasp_ranked_preview.py`
- per-arm top list from:
  - `top_candidates.left_orientation`
  - `top_candidates.right_orientation`
  - or fused equivalents if requested

With:
- `--reuse_preview_candidate_group orientation`
- `--reuse_preview_top_rank 1`

the planner uses:
- `top_candidates.left_orientation[0]`
- `top_candidates.right_orientation[0]`

for the selected frames.

## 2. Frame Rule

The preview-top1 reuse path now supports two frame modes:

- `--reuse_preview_frame_mode legacy_1_max22rel`
- `--reuse_preview_frame_mode annotated_json_keyframes`

### 2.1 Legacy mode: `legacy_1_max22rel`

If the command uses:
- `--keyframes 1 22`
- `--candidate_selection_relative_frame -10`

then frame resolution comes from preview `resolved_frames`, and the final execution frames are:
- keyframe 1 = resolved frame for `1`
- keyframe 2 = `max(resolved(22), resolved(-10))`

For the current `fi60` preview batch example:
- `resolved(-10) = 28`
- final keyframes are `1` and `28`

### 2.2 Annotated mode: `annotated_json_keyframes`

This mode is intended for:

- `/home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_direct_preview_keyframes_batch`

where preview output already contains:
- `frame 0`
- annotated keyframe 1
- annotated keyframe 2

In this mode the planner reads:
- `frame_selection.mode`
- `frame_selection.annotated_keyframes`

from preview `summary.json`.

`frame 0` remains preview-only context.

The two execution keyframes become:
- `annotated_keyframes[0]` as `grasp`
- `annotated_keyframes[1]` as `action`

So the old:
- `frame 1`
- `max(22, resolved(-10))`

is replaced by the two annotated keyframes stored in JSON.

## 3. Stage Order

The execution order is:
1. move to `pregrasp`
2. move from `pregrasp` to keyframe-1 grasp pose
3. close gripper
4. move to keyframe-2 action pose

Here "keyframe 1" means the first execution keyframe in the current frame mode:

- `1` in `legacy_1_max22rel`
- `annotated_keyframes[0]` in `annotated_json_keyframes`

`pregrasp` is not the same as that grasp keyframe.

It is computed by moving the keyframe-1 grasp pose backward along the gripper local forward axis:
- offset = `--approach_offset_m`
- current default = `0.08` m

So the effective behavior is:
- smooth EE/TCP motion to the pregrasp point
- short motion from pregrasp to keyframe-1 grasp pose
- close gripper
- smooth EE/TCP motion to keyframe-2

For the recommended path:
- `--planner_backend urdfik`
- `--urdfik_trajectory_mode cartesian_interp_ik`

the smoothing is done in EE/TCP Cartesian space first, then IK is solved waypoint by waypoint.

## 4. Object And Gripper Relation

The gripper does not automatically "follow the object" during selection.

Selection side:
- the gripper pose is chosen from AnyGrasp preview top1
- object identity is only used as a filtering / labeling constraint
- at this stage there is no physical or rigid grasp relation yet

Execution side before close:
- during `pregrasp` and `grasp`, objects stay at the replayed keyframe-1 world pose
- the gripper moves independently toward the selected AnyGrasp pose

Execution side after close:
- if `--replay_objects_during_action 0`:
  - the selected object is rigidly attached to the executing TCP during `action`
  - non-selected objects remain at their fixed replay pose
- if `--replay_objects_during_action 1`:
  - objects are replayed from `multi_object_world_poses.npz`
  - they are not attached to the TCP

Single-arm default:
- one arm executes
- the selected object for that arm is the only one that can become TCP-attached in `action` when replay mode is off

Dual-arm synchronized mode:
- both selected objects can become attached to their respective TCPs after close when replay mode is off

## 4.1 Why The Annotated Mode Exposed More "advance without reaching" Cases

The root cause was a global stage-transition bug, not a bug unique to `annotated_json_keyframes`.

Previously:
- `execute_stage_until_reached()` correctly returned `reached=False`
- but the main execution flow still continued into later stages
- so the pipeline could still go on to:
  - `close_gripper`
  - `action`

even when `pregrasp` or `grasp` had not actually reached tolerance.

Why this showed up more often in annotated mode:
- `legacy_1_max22rel` usually uses a conservative frame-1 grasp that is closer to the initial robot state
- `annotated_json_keyframes` often uses manually chosen action-heavy frames whose grasp pose is harder to reach
- so annotated mode more often produces:
  - reachable plans with final error still above tolerance
  - or grasp poses that are simply harder for the robot to realize precisely

So the apparent difference was:
- annotated mode selected harder poses more often
- the historical default still limited each stage to only `3` tries
- and the previous temporary patch made stage termination too aggressive

The current behavior is now:
- `try=3` came from `--max_stage_replans 3`
- the planner now defaults to `--replan_until_reached 1`
- it now defaults to `--replan_until_reached_max_attempts 0`
- when `replan_until_reached_max_attempts <= 0`, the retry budget is unbounded
- so if `reach=False`, the planner keeps replanning from the current state and re-executing until `reach_pos_tol_m / reach_rot_tol_deg` is satisfied
- the previous hard abort behavior after a failed `pregrasp` or `grasp` has been removed

## 5. Viewer Mode

This path also supports headed execution.

Useful arguments:
- `--enable_viewer 1`
- `--viewer_frame_delay 0.02`
- `--viewer_wait_at_end 1`

This does not change the selection logic. It only enables interactive display while the same execution pipeline runs.

## 5.1 Pure Output Mode

New argument:
- `--pure_scene_output 1`

Effect on:
- `head_cam_plan.mp4`
- `third_cam_plan.mp4`

These main output videos become clean scene renders:
- no overlay text
- no colored candidate grippers
- no target-axis visualization

Only the robot and the scene remain visible.

This does not remove debug artifacts themselves:
- `debug_selection_preview.mp4`
- `debug_execution_preview.mp4`

Those debug videos can still keep their visualization for troubleshooting.

Recommended combination:
- `--pure_scene_output 1`
- `--save_debug_preview 1`
- `--save_debug_execution_preview 1`
- `--arm auto`
- `--execute_both_arms 1`

That gives you both:
- clean main videos for scene-only review
- debug videos with axes, grasp visuals, and stage information

The default mode is now also changed to:
- `--arm auto`
- `--execute_both_arms 1`

So the planner now prefers synchronized dual-arm execution by default, and only falls back to single-arm execution if one arm has no valid selected keyframes.

## 5.2 `legacy_1_max22rel` Execution Flow

When using:
- `--reuse_preview_frame_mode legacy_1_max22rel`
- `--keyframes 1 22`
- `--candidate_selection_relative_frame -10`

the planner does:
- reads `resolved_frames` from preview `summary.json`
- resolves the second execution frame as `max(22, resolved(-10))`
- if `--reuse_preview_top_rank 1` is used, it reads the top1 candidate directly for those frames
- computes `pregrasp` as grasp pose shifted backward by 8 cm along the gripper local forward axis
- runs the stage sequence `init -> pregrasp -> grasp -> close_gripper -> keyframe_2`
- if backend is `urdfik` with `--urdfik_trajectory_mode cartesian_interp_ik`, each stage first expands into `--urdfik_cartesian_interp_steps` intermediate EE/TCP targets and solves IK along that path
- after each stage execution, it evaluates:
  - `pos_err_m`
  - `rot_err_deg`
- historical behavior was a hard cap of `3` tries
- current default behavior is to keep replanning from the current state until the stage is actually reached

## 5.3 Comparison With `annotated_json_keyframes`

The main difference between the two modes is now only the keyframe source:
- `legacy_1_max22rel`: `1 + max(22, resolved(-10))`
- `annotated_json_keyframes`: `frame_selection.annotated_keyframes[:2]`

Everything else is shared:
- both read top1 directly from preview `summary.json`
- both compute `pregrasp`
- both run `init -> pregrasp -> grasp -> close_gripper -> keyframe_2`
- both check reach after every stage
- both keep replanning on `reach=False` until the target is actually reached

## 7. Parameter Guide

Path inputs:
- `anygrasp_batch_results`: raw AnyGrasp result root
- `replay_m_obj_pose_d_pour_blue_norobot`: replay object-track root
- `hand_vis`: directory containing `hand_detections_<id>.npz`
- `anygrasp_plan_keyframes_from_preview_fi60`: planner output root
- `--reuse_preview_summary_root`: preview output root; the planner looks for `<video_name>/summary.json`

Candidate-source controls:
- `--reuse_preview_frame_mode legacy_1_max22rel|annotated_json_keyframes`
  - `legacy_1_max22rel`: keep the old `1 + max(22, resolved(-10))`
  - `annotated_json_keyframes`: use the first two entries from `frame_selection.annotated_keyframes`
- `--reuse_preview_candidate_group orientation|fused`
  - `orientation`: use the orientation-ranked preview list
  - `fused`: use the fused-score preview list
- `--reuse_preview_top_rank 1`
  - use top1
  - change to `2` for top2

Keyframe controls:
- `--keyframes 1 22`: base two-frame request
- `--candidate_selection_relative_frame -10`: add a relative frame
- final second keyframe becomes `max(22, resolved(-10))`
- these two only apply to `legacy_1_max22rel`
- under `annotated_json_keyframes`, execution keyframes come directly from `frame_selection.annotated_keyframes[:2]`
- `--candidate_selection_mode top_score_auto`: keeps auto-arm selection semantics aligned with score-first behavior

Motion controls:
- `--planner_backend urdfik`: recommended backend
- `--urdfik_trajectory_mode cartesian_interp_ik`: smooth in EE/TCP Cartesian space first, then solve IK
- `--urdfik_cartesian_interp_steps 30`: larger means smoother and slower
- `--approach_offset_m 0.08`: defines pregrasp as 8 cm behind keyframe 1 along the gripper forward axis

Execution-mode controls:
- `--arm auto`: evaluate both arms and pick the executing arm automatically
- `--arm left|right`: force a specific arm
- `--execute_both_arms 1`: synchronized dual-arm execution
- `--execute_both_arms 0`: single-arm execution

Object-relation controls:
- `--replay_objects_during_action 0`: selected object becomes TCP-attached after close
- `--replay_objects_during_action 1`: objects continue replay motion and are not attached to the TCP

Debug/display controls:
- `--pure_scene_output 1`
  - makes `head_cam_plan.mp4` / `third_cam_plan.mp4` clean scene-only outputs
- `--save_debug_preview 1`: save keyframe-selection preview video
- `--save_debug_execution_preview 1`: save slow execution preview video
- `--enable_viewer 1`: enable headed viewer
- `--viewer_frame_delay 0.02`: viewer pacing
- `--viewer_wait_at_end 1`: keep the viewer open after the run
