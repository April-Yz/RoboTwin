# V2.0 Piper Dual UR-Style Setup

## Goal

Provide a real Piper-specific AnyGrasp planning entry without modifying any R1 / R1 Pro files.

## Key idea

This V2 implementation follows the same structural idea as the existing UR setup:

- use `dual_arm_embodied=false`
- load **two independent single-arm Piper URDFs**
- space them apart using `embodiment_dis`
- keep left/right base poses separate during replay and URDF-IK execution

This is different from the earlier compatibility wrapper, which still reused too much R1-specific replay behavior.

## Robot layout

Configured in `robot_config_Piper_dual_v2.json`:

- `left_robot_file = ./assets/embodiments/piper`
- `right_robot_file = ./assets/embodiments/piper`
- `dual_arm_embodied = false`
- `embodiment_dis = 0.80`
- base poses before left/right split:
  - `[0.0, -0.65, 0.72, 1.0, 0.0, 0.0, 0.0]`
- effective base poses after `Robot(...)` applies the UR-style split:
  - left: `[-0.4, -0.65, 0.72]`
  - right: `[+0.4, -0.65, 0.72]`

So V2 uses two tabletop-mounted Piper arms, fixed in the world frame, similar in spirit to the UR dual-single-arm setup.

## Renderer behavior

`replay_piper_dual_h5.py` fixes the main issue from the earlier wrapper:

- it does **not** collapse both arms back to one shared root pose
- it stores:
  - `_left_base_pose`
  - `_right_base_pose`
- it provides arm-specific world/base conversion:
  - `world_pose_to_base_pose_for_arm(...)`
- it uses a fixed overview head camera pose instead of R1 head-link assumptions

## URDF IK behavior

`render_hand_retarget_piper_dual_npz_urdfik.py` replaces the R1-specific URDFIK assumptions:

- URDF: `assets/embodiments/piper/piper.urdf`
- base link: `base_link`
- ee link: `link6`
- 6-DoF arm seed / result handling
- no R1 torso-joint assumptions
- Cartesian waypoint IK works in each arm's own base frame
