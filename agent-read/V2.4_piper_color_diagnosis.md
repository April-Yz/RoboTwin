# V2.4 Piper Color Diagnosis: DAE Gray vs URDF Override

## User-observed state

The user reported a stable reproduction pattern:

- original Pika gripper alone: white
- original Piper arm alone: gray
- combined `piper_pika`: white / washed out

No code change was requested in this step. The goal here is diagnosis and documentation only.

## Diagnosis conclusion

Yes — the current evidence strongly supports the following explanation:

- the original Piper arm gray appearance comes from the Piper DAE files themselves
- when Piper is used inside the combined `piper_pika.urdf`, that DAE-derived gray appearance is likely overridden or washed out by explicit URDF material color blocks still present on the Piper arm links

In short:

- **original Piper gray = from DAE**
- **combined Piper whitening = likely from URDF material override in `piper_pika.urdf`**

## Source evidence

### 1. Piper DAE contains intrinsic dark gray material

Example from:

- `/home/zaijia001/Downloads/agx_arm_urdf/piper/meshes/dae/link1.dae`

The Collada material contains:

```xml
<diffuse>
  <color sid="diffuse">0.113725 0.113725 0.113725 1</color>
</diffuse>
```

This corresponds to approximately:

- RGB ≈ `(29, 29, 29)` on a 0–255 scale

So the original source asset itself is indeed a dark gray material, not a white one.

### 2. Combined URDF still contains explicit light material colors on Piper arm links

From:

- `/home/zaijia001/ssd/RoboTwin/assets/embodiments/piper_pika/piper_pika.urdf`

Example on multiple Piper arm links:

```xml
<material>
  <color rgba="0.792156862745098 0.819607843137255 0.933333333333333 1" />
</material>
```

This corresponds to approximately:

- RGB ≈ `(202, 209, 238)`

That is a very light bluish-gray tone.

Therefore the combined model currently mixes:

- DAE visual meshes from Piper
- explicit light URDF material blocks inherited from the combined reference model

This is the most likely reason why the combined Piper arm no longer looks like the original darker gray Piper.

## Interpretation of the comparison

1. Dark lighting makes the original standalone Piper closer to its DAE dark-gray source than bright lighting does.
2. Even under dark lighting, the combined model remains much farther from the DAE reference than the standalone original Piper.
3. This supports the diagnosis that the combined whitening is not only a lighting issue.
4. The explicit URDF material override is the most likely dominant cause.

## Important limitation

The PNG-based color numbers above are only approximate because they depend on:

- lighting intensity and direction
- shadows
- camera tone response
- background / segmentation error
- specular and shading behavior in the renderer

So the most trustworthy source of truth remains:

- the DAE material values themselves
- and the URDF material values that may override them

## How to visualize Piper's original color with minimal loss

Strictly speaking, a lit 3D render is never fully "lossless" with respect to source material color, because lighting and shading always affect the final pixels.

The best options are:

### Option A — Treat the DAE material values as the ground truth
For color diagnosis, read the DAE `<diffuse><color>` values directly.

This is the closest thing to a lossless source-color inspection.

### Option B — Make a flat diagnostic render
Use a diagnostic renderer/view mode with:

- no URDF material override
- neutral background
- no HDRI / no environment lighting
- minimal single white light, or ideally unlit / flat shading
- no tone-mapping or exposure tricks

This gives a much more faithful visual approximation of the source color.

### Option C — Load the DAE directly in Blender for material inspection
For the most controllable inspection:

- import the original Piper DAE into Blender
- view in a flat / material-preview style viewport
- disable aggressive environment lighting
- use Standard color management, fixed exposure

That is usually the cleanest human-readable visualization workflow.

## Recommended next action if the goal is accurate combined appearance

If a later step wants the combined robot to preserve the original Piper gray while keeping the white Pika gripper, the likely fix will be:

- remove the explicit light `<material><color ...>` blocks from the Piper arm links in `piper_pika.urdf`
- keep the Pika gripper white material blocks as-is
- re-render and compare again
