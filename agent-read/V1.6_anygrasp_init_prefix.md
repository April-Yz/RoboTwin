# V1.6 AnyGrasp Init Prefix Start

Date: 2026-03-17

## What Changed

- File: `code_painting/plan_anygrasp_keyframes_r1.py`
- Added CLI argument:
  - `--init_prefix_frames` (int, default `0`)
- Added explicit init-pose reset step before stage execution:
  - Re-apply renderer init joints (`init_left_arm_joints`, `init_right_arm_joints`) if available.
  - Re-apply init gripper openness (`init_gripper_open`, fallback to `--open_gripper`).
- Added optional fixed-prefix frame emission:
  - Before moving to keyframe-1, write the first `N` frames as constant init-pose frames.
  - Overlay label uses `stage=init_prefix` and `fixed_frame=i/N`.

## Why

- User requested that planning/execution starts from init pose first, then move to frame-1 goal.
- User also requested a fixed-prefix-frame mechanism so these leading frames are easy to trim/delete later in post processing.

## Affected Modes

- Single-arm execution path: each executed arm sequence now starts from init pose and can emit init-prefix frames.
- Dual-sync execution path (`--arm auto --execute_both_arms 1`): synchronized run starts from init pose and emits shared init-prefix frames.

## Output / Summary Changes

- `plan_summary.json` now includes:
  - `init_prefix_frames`
  - `init_pose_by_executed_arm`
  - `init_prefix_frames_written_by_executed_arm`

## Usage Notes

- Keep existing behavior: do nothing extra by default (`--init_prefix_frames 0`).
- To force a 12-frame static prefix at init pose:
  - `--init_prefix_frames 12`
- This does not remove existing planning stages; it only prepends fixed init frames before stage motion begins.
