## 1. 先说结论

这里其实有两套“排名”概念，必须分开看：

1. `render_anygrasp_ranked_preview.py` 的当前逻辑
- 现在不再只是单一 `score rank` 预览，而是三阶段预览工具。
- 第 1 阶段：先按最近物体做一次共享划分，并在终端输出每个物体各拿到多少个候选。
- 然后再通过 `left_target_object/right_target_object` 把这些物体桶映射到左右手。
- 第 2 阶段：在物体筛选后的集合上，只按朝向相似度做排序，输出一张左右合并图。
- 第 3 阶段：在同一集合上按融合分数排序，默认是 `0.5 * anygrasp_score + 0.5 * orientation_score`，再输出一张左右合并图。
- 在第 2/3 阶段排序之前，staged 模式现在还会默认过滤掉和人手朝向旋转差大于 `90` 度的候选。
- 预览图现在还会用另一种颜色把“人手参考姿态本身”画出来，这样可以直接在同一张图里对比 AnyGrasp 候选和人手参考朝向是否一致。
- 不再保存单独的左右手图片，只保存左右综合图。

1a. `run_render_anygrasp_ranked_preview_batch.sh`
- 这是这条预览链路的 batch 包装层。
- 它会自动循环 `d_pour_blue_{id}` 目录，组装每个视频对应的 `anygrasp_dir/replay_dir/hand_npz/base_image_dir/output_dir`，然后把剩余参数继续转发给 `render_anygrasp_ranked_preview.py`。

2. `plan_anygrasp_keyframes_r1.py` 的排名
- 这是实际决定 RoboTwin 最终抓哪个候选、哪个手来执行的逻辑。
- 它先把候选映射到世界坐标，再给每个候选匹配最近物体，再做目标物体过滤和距离过滤。
- 过滤后的候选按
  - `rotation_distance_deg` 升序
  - `nearest_object_distance_m` 升序
  - `score` 降序
  排序。
- 然后再在两个关键帧之间选一个“同一物体”的组合，代价最小者胜出。

所以：

- 预览图里的 `score rank` 不是最终执行 rank。
- 真正的执行顺序更偏向“手姿态更接近 + 候选更靠近目标物体”，而不是单纯看 AnyGrasp 分数。

## 3. `render_anygrasp_ranked_preview.py` 的当前逻辑

## 3.1 作用

这个脚本仍然是一个“不跑正式执行器的预览工具”，但它现在会模拟 planner 里的前几层筛选逻辑。

它的目标变成了：

1. 先看目标物体约束会把候选裁成什么样。
2. 再看只按手朝向时左右手各自的排序。
3. 最后看原始 AnyGrasp 分数和朝向分数加权后的排序。

这份文档里说的 preview 阶段 “Select”，指的是每一帧里的完整筛选链：
1. 共享最近物体划分
2. 通过 `left_target_object/right_target_object` 做左右手映射
3. 用对齐后的人手朝向做 `max_rotation_distance_deg` 过滤
4. 导出 orientation/fused 的 top-N 图片和 JSON 摘要

## 3.2 输入

主要输入：

- `grasps/grasp_<frame>.json`
- `hand_detections_<id>.npz`
- 可选底图：
  - replay 导出的 `color_<frame>.png`
  - 或 AnyGrasp 自己的 `vis/grasp_result_<frame>.png`

额外前提：

- 如果传了 `--replay_dir`
  - 脚本会启用 staged 模式
  - 物体筛选需要 `replay_dir/multi_object_world_poses.npz`
  - 优先使用里面的 `head_camera_pose_world_wxyz`
  - 如果旧 replay 没有这个字段，脚本现在会自动 fallback：
    - 默认直接使用脚本内置的固定 `head_camera_pose_world_wxyz`
    - 也可以手动通过 `--fallback_head_camera_pose_world_wxyz` 覆盖
  - 这个 fallback 不再启动模拟器
- 如果不传 `--replay_dir`
  - 脚本会退回到旧的 raw AnyGrasp score-only 综合图模式

## 3.3 第 1 阶段：物体筛选

当前核心函数是：

- `load_replay_frame_data(...)`
- `nearest_object_name_in_camera(...)`
- `build_candidates_for_arm(...)`

处理顺序是：

1. 读取 replay 导出的物体 world pose
2. 读取同一帧的 `head_camera_pose_world_wxyz`
3. 用和 planner 一致的 `legacy_r1` 相机轴变换，把 camera-frame 的 grasp target 先转到 world
4. 在 world 系里计算每个 grasp 到每个 visible object 的欧氏距离
5. 取最近物体 `nearest_object`
5. 左手只保留 `nearest_object == left_target_object`
6. 右手只保留 `nearest_object == right_target_object`

注意：

- “左右手怎么区分物体”这一步本身不是按手来算距离。
- 每个 candidate 先统一得到一个 `nearest_object`。
- 左右手的区别是在下一步按 `left_target_object/right_target_object` 把共享物体桶映射到左右手，以及后面的手朝向排序。

如果手动打开 `--draw_object_overlay 1`，staged 预览图还会把这些 replay 物体直接画出来：

- 物体中心点
- 物体名称小标签

终端会输出：

- 总候选数
- 左手筛选前后数量
- 右手筛选前后数量

格式类似：

```text
[frame 000027][object-filter] before_total=60 left_before=60 right_before=60 left_after=12 target=cup right_after=9 target=bottle
```

## 3.4 第 2 阶段：朝向排序

在物体筛选后的候选上，脚本会计算：

- `rotation_distance_deg`
- `orientation_score = 1 - rotation_distance_deg / 180`

然后按下面的 key 排序：

```python
(-orientation_score, rotation_distance_deg, -anygrasp_score)
```

输出文件：

- `frame_<frame>_left_right_orientation_rank.png`

每一行会显示：

- `idx`
- `ori`
- `rot`
- `obj`

## 3.5 第 3 阶段：融合分数排序

在同一个物体筛选后集合上，脚本还会计算：

```python
fused_score = anygrasp_score_weight * anygrasp_score + orientation_score_weight * orientation_score
```

默认就是：

```python
0.5 * anygrasp_score + 0.5 * orientation_score
```

排序 key 是：

```python
(-fused_score, -anygrasp_score, -orientation_score)
```

输出文件：

- `frame_<frame>_left_right_fused_rank.png`

面板里会直接把两项分数和总分写出来，格式类似：

```text
0.812*0.50 + 0.744*0.50 = 0.778
```

## 3.6 `summary.json` 里的头相机字段

在 staged 模式下，`summary.json` 现在会额外记录：

- `camera_pose_world_wxyz`
- `camera_pose_source`

其中 `camera_pose_source` 可能是：

- `replay_npz`
- `fixed_fallback`

另外如果打开：

- `--debug_dump_object_distances 1`

每一帧还会额外导出：

- `frame_<frame>_object_distance_debug.json`

里面会记录：

- candidate 的 camera/world 位置
- 每个 visible object 的 world 位置
- candidate 到每个 object 的 world 距离
- 最终 `nearest_object`

默认情况下，staged 模式和 score-only 模式会统一使用同一类底图逻辑：

- 都直接在 `base_image` 上叠加夹爪可视化
- staged 模式默认不再额外画 obj overlay
- 只有显式传 `--draw_object_overlay 1` 时，才会把 replay 物体标出来

## 4. 真正的 planner 排名逻辑

真正的抓取候选排序在 [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py) 里。

最核心的三层函数是：

- `build_ranked_candidates_for_arm(...)`
- `select_keyframes_for_arm(...)`
- `choose_keyframes(...)`

### 4.1 单个候选先被转换成世界系目标

在 `candidate_to_world_pose(...)` 里，每个 AnyGrasp 候选先经过：

1. 从相机系读取 `rotation_matrix` 和 `translation`
2. 应用
   - `candidate_post_rot_matrix`
   - `candidate_orientation_remap_matrix`
3. 用 `renderer.camera_to_world_pose(...)` 变成世界系 pose
4. 如果设置了 `--candidate_target_local_x_offset_m`，再沿局部 `+X` 平移
5. 如果开了 `--candidate_keep_camera_up 1`，再在“原姿态 / 绕局部 +X 翻 180 度”之间选更朝上的版本

这一步并不决定 rank，但会改变最终参与规划的姿态。

### 4.2 每个候选先匹配最近物体

在 `nearest_object_name(...)` 里：

- 用候选世界坐标位置和每个可见物体的世界坐标位置做欧氏距离
- 取最近的物体名 `nearest_object`
- 以及对应距离 `nearest_object_distance_m`

这一步为后续约束提供标签。

### 4.3 每只手、每个关键帧的候选排序

核心函数：

- `build_ranked_candidates_for_arm(...)`

对每个 `frame`，它的顺序是：

1. 找这只手在该帧的参考姿态
- 优先用当前帧 `hand_rotations[frame]`
- 若当前帧无效，就回退到最近有效手帧

2. 遍历 AnyGrasp 所有候选
- 读取 `score`
- 计算世界系 pose
- 匹配最近物体与最近距离
- 计算 `rotation_distance_deg`

3. 做目标物体过滤
- 若 `--enforce_target_object_constraint 1`
- 左手只保留 `nearest_object == left_target_object`
- 右手只保留 `nearest_object == right_target_object`

4. 做候选距离过滤
- 若 `--enforce_candidate_distance_constraint 1`
- 只保留 `nearest_object_distance_m <= candidate_object_max_distance_m`

5. 对候选排序

代码是：

```python
all_candidates.sort(key=lambda cand: (cand.rotation_distance_deg, cand.nearest_object_distance_m, -cand.score))
```

同时它还会对 `raw_candidates` 用同样的 key 排一次：

```python
raw_candidates.sort(key=lambda cand: (cand.rotation_distance_deg, cand.nearest_object_distance_m, -cand.score))
```

但要注意命名：

- `raw_candidates`
  - 是“不过滤 object/distance 的全集”
  - 不是“按 raw score 排序”
- `all_candidates`
  - 是“通过 object/distance 过滤后的候选”
  - 才是该手实际用于后续选择的 ranked list

所以这里真正的 rank key 是：

1. 先最小化手姿态差 `rotation_distance_deg`
2. 再最小化候选到最近物体中心的距离 `nearest_object_distance_m`
3. 最后才用 AnyGrasp `score` 作为第三排序键

这和 `render_anygrasp_ranked_preview.py` 的 score-only 逻辑是根本不同的。

## 5. 两个关键帧之间怎么选最终组合

单帧有 rank 之后，还没有得到最终执行的两个 keyframe 候选。最终组合发生在：

- `select_keyframes_for_arm(...)`

### 5.1 手工覆盖优先级最高

先看 `--manual_candidate`：

- `parse_manual_candidate_overrides(...)` 解析用户指定
- `reorder_candidates_with_manual_choice(...)` 会把指定的 `candidate_idx` 挪到列表第一个

如果某只手在两个关键帧都被手工指定了 candidate：

- planner 直接采用这两个 candidate
- 不再跑自动组合搜索

如果只指定了部分帧：

- 只影响展示顺序和候选优先级
- 自动选择逻辑仍然继续执行

### 5.2 自动组合要求两个关键帧抓同一个物体

自动模式里，planner 会：

1. 遍历第一关键帧的每个候选 `first_candidate`
2. 取它的 `nearest_object`
3. 只在第二关键帧里找 `nearest_object` 相同的候选

也就是：

- 第一帧和第二帧必须对应同一个物体

### 5.3 自动组合代价函数

代码里的总代价是：

```python
total_cost = (
    first_candidate.rotation_distance_deg
    + best_second.rotation_distance_deg
    + 10.0 * (first_candidate.nearest_object_distance_m + best_second.nearest_object_distance_m)
)
```

这意味着：

1. 两帧的手姿态误差都越小越好
2. 两帧的候选离对应物体都越近越好
3. 物体距离项乘了 `10.0`，说明它在总代价里被显式放大
4. 这个阶段不再显式使用 `score`

为什么第二帧直接取 `second_candidates[0]`：

- 因为第二帧候选已经在前一步按
  - `rotation_distance_deg`
  - `nearest_object_distance_m`
  - `-score`
  排好序了
- 所以同一物体下的第一个候选，就是这个物体在第二关键帧里的最优候选

## 6. 左右手之间怎么决定谁执行

在 `choose_keyframes(...)` 里：

- 如果 `--arm left/right`，只评估那只手
- 如果 `--arm auto`，会分别求左手和右手的最佳两关键帧组合

然后比较：

```python
total_cost = sum(item.candidate.rotation_distance_deg for item in selection.selected_keyframes)
```

也就是左右手最终竞争时，比较的是：

- 两个已选关键帧的 `rotation_distance_deg` 之和

这里没有再把 `nearest_object_distance_m` 加进去。原因是：

- 每只手内部在选两关键帧组合时，已经用 `distance` 把明显不贴物体的解压下去了
- 到左右手竞争这一步，只用旋转匹配度决定谁更像当前手的示范姿态

## 7. 为什么预览图和最终选择可能不一致

最常见的原因有四类：

1. 当前预览脚本只模拟到“单帧物体筛选 + 朝向排序 + 融合分数排序”
2. planner 还会继续做 `rotation -> object distance -> score` 的正式排序
3. planner 还会继续做双关键帧“同一物体”组合搜索
4. `--arm auto` 下 planner 还会继续在左右手之间选总代价更小的一只手

因此可能出现：

- 某个候选在 score 预览图里是 top-1
- 但因为手姿态差太大、离目标物体太远、或者和第二关键帧对不上同一物体
- 最终不会被执行

## 8. 几个容易混淆的数据结构

### 8.1 `ranked_candidates_per_frame`

- 来自 `build_ranked_candidates_for_arm(...)` 的过滤后候选
- 顺序是最终 planner 真正参考的自动排序顺序
- 会被 `manual_candidate` 做“挪到最前”处理

### 8.2 `all_candidates_per_frame`

- 实际装的是未经过 object/distance 过滤的全集
- 但仍然按 `rotation_distance_deg, nearest_object_distance_m, -score` 排过序
- 主要用于 debug 展示和手工检查

### 8.3 `common_candidates_per_frame`

- 来自 `build_display_candidates_per_frame(...)`
- 如果 `debug_show_all_candidates=1`，它会把 `all_candidates_per_frame` 再按 `score` 降序重排后显示成绿色 common candidates

这一点很关键：

- 绿色 common candidate 的显示顺序，又回到了纯 `score` 视角
- 但蓝色/橙色的 arm-specific top-k，来自 `ranked_candidates_per_frame`
- 所以一个画面里本来就同时混着两种排序语义

两者名字接近，但不是一回事：

## 10. 一句话总结

一句话说完这套逻辑：

- `render_anygrasp_ranked_preview.py` 现在做的是 “先做共享物体划分，再做左右手映射，再看朝向排序，再看 `AnyGrasp + 朝向` 的融合排序”
- `plan_anygrasp_keyframes_r1.py` 真正执行的则更深一层：还会继续按物体距离、双关键帧同物体组合、以及左右手自动选择去做最终决策

如果要人工挑 candidate：

- 先看 `render_anygrasp_ranked_preview.py` 了解原始 score 候选分布
- 再看 planner 输出的 `rank_previews/` 和 `plan_summary.json`
- 因为真正执行顺序以后者为准

## 11. 人手坐标轴和 AnyGrasp 坐标轴的区别

这里最容易混淆的一点是：人手检测和 AnyGrasp 预览线框，并不共享同一个“局部前向轴”定义。

`detect_hands_realr1.py` 里的人手局部坐标是这样构造的：
- 局部 `+Y`：`thumb_tip - index_tip`，也就是开合方向
- 局部 `+X`：`cross(index_joint - index_tip, y_axis)`
- 局部 `+Z`：`cross(x_axis, y_axis)`

同时它把 retreat 定义成：
- `wrist_position_retreat = gripper_position - retreat_distance * z_axis`

所以在人手检测这边，局部 `+Z` 实际上更像“接近/退让轴”。

但预览图里复用的 AnyGrasp 线框函数默认是：
- 局部 `+X`：手指前向轴
- 局部 `+Y`：开合轴

这就是为什么如果直接拿人手的 `rotation_matrix` 去套 AnyGrasp 线框，视觉上会出现“本来朝前的人手，看起来像朝下”这种错觉。
