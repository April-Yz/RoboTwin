# V1.10 AnyGrasp Camera-Up Rule

## What Changed

Added a planner-side orientation post-process for AnyGrasp candidates:

- `--candidate_keep_camera_up 1`
- `--candidate_camera_top_axis z|y`

When enabled, the planner keeps the original grasp direction and only resolves the redundant roll about the gripper forward axis conservatively. It prefers a candidate orientation whose chosen local "top axis" stays on the upward side of world `+Z`, instead of forcing an exact alignment.

## Why This Was Needed

The user observed that keyframe-1 and keyframe-22 could differ by almost a full `360` degree roll while still representing the same grasp for the gripper. This is expected because the grasp is largely defined by:

- the TCP position
- the forward/approach axis
- the jaw opening axis

Rotation around the forward axis is often redundant for grasp execution, but it changes where the camera mounted on the gripper ends up. For the user's setup, the desired behavior is:

- keep the mounted camera on top
- do not allow arbitrary roll around the gripper forward axis

## Axis Analysis

The current planner and robot chain imply:

- local `+X` is the gripper forward / approach axis

Evidence in code:

- `pose_with_offset_along_local_x(...)` in [`code_painting/plan_anygrasp_keyframes_r1.py`](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py)
- `_trans_from_gripper_to_endlink(...)` in [`envs/robot/robot.py`](/home/zaijia001/ssd/RoboTwin/envs/robot/robot.py)

So the ambiguous degree of freedom is:

- roll around local `+X`

The new rule keeps local `+X` fixed and only resolves that roll.

## New Rule

For each AnyGrasp candidate:

1. Convert candidate camera-space rotation to world-space as before.
2. If `--candidate_keep_camera_up 1` is enabled:
   - keep local `+X` unchanged
   - compare:
     - the original candidate orientation
     - the same orientation rolled by `180` degrees around local `+X`
   - choose whichever one gives a larger upward component for the configured top axis

This means:

- approach direction is preserved
- the planner does not snap roll to one exact value
- side grasps keep their original look unless they are upside down
- the gripper camera/top stays on the upward side overall

This is intentionally conservative. It fixes upside-down roll ambiguity without over-constraining sideward grasps.

## New Parameters

- `--candidate_keep_camera_up 1`
  - enable roll disambiguation
- `--candidate_camera_top_axis z`
  - treat local `+Z` as the gripper top / camera-up axis
- `--candidate_camera_top_axis y`
  - alternative if later inspection shows the mounted camera is aligned with local `+Y`

## Summary Output

`plan_summary.json` now records:

- `candidate_keep_camera_up`
- `candidate_camera_top_axis`
- `top_axis_up_dot` for selected and ranked candidates

`top_axis_up_dot` is the dot product between the selected top axis and world `+Z`:

- close to `1.0`: top axis points upward
- close to `0.0`: top axis is horizontal
- negative: top axis points downward

Additional debug fields:

- `original_top_axis_up_dot`
- `camera_up_flip_applied`
- `forward_axis_change_deg`

Expected behavior:

- `forward_axis_change_deg` should stay `0.0`
- `camera_up_flip_applied` is `1` only when the upward-facing `180`-degree roll variant was chosen

## Difference From Recent Earlier Changes

The recent commits before this feature split into two categories.

Behavior-changing:

- `d877396`: replay object tracks during the action stage
- `830f057`: allow per-object mesh override, e.g. use `blue_cup.obj`

Debug-only:

- `27a4bc9`: dump replay-side pose debug
- `6a0974f`: dump planner-side pose debug and compare export-time vs planner-time head camera

So if a run became "correct" after these changes, the direct behavior changes came from:

- action-stage object replay
- mesh override

The pose-debug commits only improved inspection and did not intentionally change planner motion logic.

## Previous Error In The First Camera-Up Attempt

The first implementation was too aggressive.

It tried to:

- keep local `+X` fixed
- force the configured top axis to align with world up as much as possible

That was mathematically valid, but it changed the visible roll too strongly. In practice it made some side grasps look wrong even though the forward axis stayed unchanged.

The corrected implementation is closer to the real user intent:

- do not pin the gripper to one exact roll
- only avoid upside-down camera placement
- preserve the original sideward grasp style whenever possible
