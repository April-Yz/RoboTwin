# V1.11 AnyGrasp Target Pose Offset

## Problem Statement

The planner-side target pose used by the AnyGrasp workflow can be ambiguous:

- the raw AnyGrasp candidate may behave like a fingertip TCP pose
- or it may behave like a wrist/endlink pose in the current RoboTwin gripper convention

This matters because the `urdfik` path does **not** solve directly on the incoming world target pose.

The current chain is:

1. `plan_anygrasp_keyframes_r1.py` builds a world-space candidate pose
2. `renderer.plan_path(arm, target_pose_world)`
3. `render_hand_retarget_r1_npz_urdfik.py::_target_tcp_world_to_ee_base(...)`
4. `robot._trans_from_gripper_to_endlink(...)`
5. solve IK for `left_gripper_link` / `right_gripper_link`

So the solver target is an endlink/EE pose, not the raw incoming pose itself.

## What The Existing Code Already Proved

The current implementation already documents that:

- the planner treats the incoming AnyGrasp target as a gripper/TCP-style pose
- `_trans_from_gripper_to_endlink(...)` then shifts it to the wrist/endlink convention
- the shift is approximately `+0.12 m` along local gripper `+X`

Relevant code:

- [`envs/robot/robot.py`](/home/zaijia001/ssd/RoboTwin/envs/robot/robot.py)
- [`code_painting/render_hand_retarget_r1_npz_urdfik.py`](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz_urdfik.py)
- [`code_painting/plan_anygrasp_keyframes_r1.py`](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py)

This means the user suspicion was valid:

- if the raw AnyGrasp candidate already behaves like a wrist/endlink pose in this setup,
- then the current TCP->EE conversion will push the IK target too far forward unless the planner target is shifted backward first.

## New Parameter

Added:

- `--candidate_target_local_x_offset_m`

Semantics:

- before planning and before target-axis visualization,
- translate each candidate target pose by this amount along the gripper local `+X` axis

Examples:

- `0.0`
  - keep the current behavior
- `-0.12`
  - move the target backward by `12 cm` along local `+X`
  - use this if the raw AnyGrasp candidate behaves like a wrist/endlink pose and you want the planner input to represent the fingertip TCP instead

## Why This Fix Is Better Than Hard-Coding One Convention

The exact meaning of the raw AnyGrasp pose depends on:

- the mesh/gripper convention used upstream
- the local definition of the grasp frame
- the RoboTwin TCP vs endlink convention

Hard-coding one offset globally would be brittle.

An explicit parameter is safer because it lets the workflow switch between:

- raw candidate pose
- fingertip-adjusted target pose

without rewriting the planner again.

## What Changed In The Planner

The planner now stores two versions of each candidate:

- `raw_pose_world_wxyz`
  - the direct AnyGrasp world pose after orientation remap / camera-up processing
- `pose_world_wxyz`
  - the actual planning / visualization target after `--candidate_target_local_x_offset_m`

So the target axis shown in the viewer now follows the actual pose sent into planning, not only the raw imported candidate.

`plan_summary.json` now records:

- top-level `candidate_target_local_x_offset_m`
- per-candidate `raw_pose_world_wxyz`
- per-candidate `pose_world_wxyz`

## Function Names And Actual Solve Target

The active planning function names are:

- planner entry used by the keyframe workflow:
  - `HandRetargetR1URDFIKRenderer.plan_path(...)`
- internal conversion:
  - `_target_tcp_world_to_ee_base(...)`
- final IK solver call:
  - `URDFInverseKinematics.solve_ik(...)`

The actual quantity being solved is:

- the target pose of `left_gripper_link` or `right_gripper_link`
- expressed in the robot base frame

It is **not** solving directly on the raw world-space AnyGrasp pose.

## Recommended Command

If the raw candidate currently looks like a wrist/endlink target and you want a fingertip TCP target instead, add:

```bash
--candidate_target_local_x_offset_m -0.12
```

Example:

```bash
CUDA_VISIBLE_DEVICES=3 bash /home/zaijia001/ssd/RoboTwin/code_painting/run_plan_anygrasp_keyframes_r1_batch.sh \
  /home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_batch_results \
  /home/zaijia001/ssd/RoboTwin/code_painting/replay_m_obj_pose_debug \
  /home/zaijia001/ssd/data/R1/gt_depth_vis/d_pour_blue/hand_vis \
  /home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_plan_keyframes \
  --ids 1 \
  --keyframes 1 22 \
  --arm auto \
  --execute_both_arms 1 \
  --planner_backend urdfik \
  --candidate_target_local_x_offset_m -0.12 \
  --candidate_keep_camera_up 1 \
  --candidate_camera_top_axis z \
  --manual_candidate 1 left 5 \
  --manual_candidate 1 right 8 \
  --manual_candidate 22 left 7 \
  --manual_candidate 22 right 1 \
  --reach_error_pose_source ee \
  --replan_until_reached 1 \
  --replan_until_reached_max_attempts 20 \
  --enable_viewer 1
```
