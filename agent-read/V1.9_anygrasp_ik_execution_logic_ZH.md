# AnyGrasp 执行链路与 IK 逻辑说明

## 1. 你原来的规划逻辑是什么

当前 `AnyGrasp -> RoboTwin` 这条执行链，在 `--planner_backend urdfik` 时，核心分成 4 步：

1. 读入 AnyGrasp 给的目标抓取位姿  
2. 把这个目标从相机坐标系转成世界坐标系，再转成机器人基座坐标系  
3. 把“夹爪 TCP 目标位姿”转换成“机械臂末端 link 目标位姿”  
4. 用 IK 求一个目标关节角，再把当前关节角插值到目标关节角，逐步执行

所以它不是：
- 直接把机械臂末端 link 瞬间设置到目标位姿

而是：
- 先解 IK
- 再把关节轨迹离散成多个中间步
- 一步一步执行

## 2. 现在的 IK 求解是不是自己写的

### 简短答案

不是“从零手写一个新的 IK 数值求解器”，而是：

- 在项目里封装了一个 `URDFInverseKinematics`
- 这个类内部实际调用的是 `curobo` 的 `IKSolver`
- 但它是按本地 R1 的 URDF、base_link、ee_link 这套配置来解的

所以更准确的说法是：

- 你现在用的是“项目内封装的、面向 R1 的 URDF IK 求解链”
- 输入目标 EE pose
- 输出目标关节角

## 3. 输入的 pose 到底是什么

不是直接拿 TCP pose 去解 endlink IK。

当前逻辑是：

1. 你给的是 gripper TCP 目标位姿
2. 代码先把 TCP pose 转成机器人 base 坐标系
3. 再调用 `_trans_from_gripper_to_endlink(...)`
4. 把 gripper pose 转成 endlink pose
5. IK 对 endlink pose 解关节角

转换位置：
- [render_hand_retarget_r1_npz_urdfik.py:82](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz_urdfik.py:82)
- [render_hand_retarget_r1_npz_urdfik.py:86](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz_urdfik.py:86)

真正的 TCP -> endlink 变换逻辑在：
- [robot.py:333](/home/zaijia001/ssd/RoboTwin/envs/robot/robot.py:333)

- 做 gripper bias 平移
- 做固定姿态矩阵修正

所以这条链不是“目标位姿直接喂给 IK”，中间确实有一层手爪到末端 link 的固定转换。

## 4. 现在的执行逻辑是什么

### `urdfik plan_path(...)` 实际输出什么

`HandRetargetR1URDFIKRenderer.plan_path(...)` 并不会生成一个复杂轨迹优化结果。

它只做：

1. 读取当前 6 维 arm joints  
2. 调 IK 解出一个目标 6 维 arm joints  
3. 构造一个只有两帧的 joint trajectory：
   - 当前关节角
   - 目标关节角

代码位置：
- [render_hand_retarget_r1_npz_urdfik.py:89](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz_urdfik.py:89)
- [render_hand_retarget_r1_npz_urdfik.py:123](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz_urdfik.py:123)

也就是说：

- `urdfik` 本身只给你“起点关节角 -> 终点关节角”
- 中间那些很多步，不是 IK 求出来的，而是后面执行器自己补出来的

## 5. 为什么中间会有 25 个关键步

因为执行时会做关节插值。

### 插值逻辑

在执行前，`position` 和 `velocity` 会先送进：
- [plan_anygrasp_keyframes_r1.py:1087](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py:1087)

也就是：
- `interpolate_joint_trajectory(...)`

如果当前轨迹点数太少，而你设置了：
- `--execute_interp_steps 24`

它就会把原本的“2 个点轨迹”插值成更多个中间关节状态。

所以视频里看到：
- `plan_step=1/25`
- `plan_step=2/25`
- ...
- `plan_step=25/25`

本质上是：
- 当前关节角到目标关节角之间的人为平滑离散步

不是：
- 25 个新的关键帧
- 25 次 IK
- 25 次重规划

## 6. 如果不加这 25 个中间步会怎样

### 逻辑上

如果不做这层插值，那么 `urdfik` 输出的轨迹通常只有：

1. 当前关节角
2. 目标关节角

执行器就只会非常粗地从当前直接切到目标。

### 在 viewer 里会发生什么

你在 viewer 里看到的效果通常会接近：

- 非常快地跳过去
- 几乎像“瞬移”

尤其在这几个条件同时满足时更明显：

- 轨迹只有 2 个点
- `viewer_frame_delay` 很小
- 每个点只 `step_scene(1)`

所以中间插值的主要目的不是“让 IK 更准”，而是：

1. 让动作更可视化
2. 让执行更平滑
3. 给 reach check 更多中间状态

## 7. 这 25 步会不会让精度更高

不一定。

它主要改善的是：
- 视觉连续性
- 动作平滑度

它不会从根本上改变：
- IK 最终目标是不是解对了
- TCP/endlink 转换是不是对的
- 目标姿态本身是不是合理

所以如果末端最终误差大，真正该查的是：

1. 目标 pose 是否合理
2. gripper TCP -> endlink 转换是否正确
3. 误差评估口径是 `tcp` 还是 `ee`
4. IK 解本身是否卡在局部解

而不是单纯把 25 改成更大。

## 8. 当前这套执行逻辑的一句话总结

当前 `urdfik` 模式的执行逻辑是：

- 输入目标 gripper TCP pose
- 转成 base 下的 endlink target pose
- 用项目里的 `URDFInverseKinematics`（内部调用 curobo IKSolver）解出目标关节角
- 构造“当前 joints -> 目标 joints”的两点轨迹
- 再用 `execute_interp_steps` 把它插值成很多中间 joint steps
- 在 viewer / 视频里逐步执行

所以你现在看到的 25 步，本质上是：

- “关节空间插值执行步数”

不是：

- “额外 25 个真正的路径关键帧”
