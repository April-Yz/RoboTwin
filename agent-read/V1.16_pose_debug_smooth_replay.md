# V1.16 Smooth replay from pose_debug.jsonl

## Purpose

When `plan_anygrasp_keyframes_r1.py` runs with:

- `--pure_scene_output 1`

it keeps the main export clean and automatically writes:

- `pose_debug.jsonl`

This file records per-frame:

- left/right arm joint positions
- gripper command and gripper joint qpos
- head camera / TCP / EE poses
- current execution-object actor poses

A new post-processing replay tool is now available:

- `code_painting/replay_pose_debug_smooth.py`
- wrapper: `code_painting/run_replay_pose_debug_smooth.sh`

Its role is:

- do not change the original planning/execution result
- re-render the recorded pose-debug trajectory with denser interpolation
- export a visually smoother replay video

## Core semantics

The new path does:

1. load `pose_debug.jsonl`
2. interpolate between adjacent records:
   - arm joint qpos: linear interpolation
   - gripper joint qpos / gripper command: linear interpolation
   - object actor poses: linear position interpolation + quaternion slerp
3. render the interpolated states into a new video

So this is:

- **post-hoc visual replay smoothing**

not:

- online control compensation
- replanning
- improving the original reach accuracy

More concretely, the replay is driven directly by the states saved in the original run. The current script uses recorded fields such as:

- `current_left_arm_qpos_rad`
- `current_right_arm_qpos_rad`
- `current_left_gripper_joint_qpos_rad`
- `current_right_gripper_joint_qpos_rad`
- `current_left_gripper_command`
- `current_right_gripper_command`
- `object_actor_poses.*.actor_pose_world_wxyz`

Then it interpolates between those already-recorded states and re-renders the scene.

So the direct answer is:

- **yes, this replay smooths and replays the original saved joint / gripper / object states directly**
- it does **not** rerun the planner or redo IK

## Alignment between saved records and video frames

`pose_debug.jsonl` is written inside `record_frame(...)` in `plan_anygrasp_keyframes_r1.py`.

That means:

- every time `record_frame(...)` is called
  - one video frame is written
  - one `pose_debug.jsonl` record is appended
- so the records are aligned with the **frames that were actually exported**
- each record can contain fields such as:
  - `current_left/right_arm_qpos_rad`
  - `current_left/right_gripper_joint_qpos_rad`
  - `current_left/right_gripper_command`
  - `current_left/right_tcp_pose_world_wxyz`
  - `current_left/right_ee_pose_world_wxyz`
  - `current_head_camera_pose_world_wxyz`
  - `object_actor_poses`

But an important caveat is:

- this is **not** one record per physics step
- it is only one record per explicit `record_frame(...)` call

So the correct statement is:

- **aligned with exported video frames: yes**
- **aligned with every low-level physics substep: no**

This is also why some motion can still look sparse in the original output:

- the command loop usually records one frame per commanded waypoint
- settle / wait phases are not recorded at every physics substep either

## Important limitation

This tool improves:

- video appearance
- presentation smoothness

It does **not** change:

- the original run's reach result
- the original run's `plan_summary.json` errors
- the original controller convergence behavior

In short:

- it makes the replay video smoother
- it does not make the original execution more accurate

## Recommended use case

Use it when:

- a pure-mode run already finished
- you want a smoother presentation video from that exact run
- you do not want to re-run AnyGrasp / planning / execution

Do not use it as a substitute for:

- controller-follow analysis
- reach / attempt / pos_err / rot_err debugging

## Key parameters

- `--plan_summary_json`
  - recommended entry; infers `pose_debug.jsonl` and `replay_dir`
- `--pose_debug_jsonl`
  - direct replay input
- `--replay_dir`
  - used to load replay object meshes
- `--output_path`
  - output video path
- `--interp_factor`
  - number of replay subframes inserted per adjacent recorded pair
  - `1` means no extra interpolation
  - recommended starting value: `3` or `4`
- `--fps`
  - output video FPS
- `--overlay_text`
  - whether to draw lightweight labels onto the smoothed replay video
- `--max_input_records`
  - replay only the first N records; useful for smoke tests

## What the smoothing actually does

The smoother does not modify the planner's original 20 Cartesian waypoints, and it does not rerun the controller.

What it actually does is:

- treat the already-saved `pose_debug.jsonl` records as keyframes
- insert additional subframes between adjacent saved records
- re-render a denser replay video from those interpolated states

So it is:

- **record-to-record interpolation / frame-interval smoothing**

not:

- recovering every missing low-level controller substep

If you feel that the original `20` interpolation waypoints still did not look very smooth, there are usually two possible reasons:

### Case A: the real motion was not recorded densely enough in the first place

This is the most common case.

For example, in the original execution path:

- one frame is usually recorded per commanded waypoint
- only a small number of physics steps may happen inside each waypoint command
- there can also be later `settle_steps` / `joint_target_wait_steps` convergence phases
- those phases are not serialized into the main video at every physics substep either

In that situation, `pose_debug.jsonl` itself is already a sparse sequence of key states.

So post-hoc smoothing can only:

- create a visually smoother transition between those sparse saved states

It cannot recover:

- motion details that were never recorded in the first place

### Case B: the current smoothing model is intentionally simple

The current smoother uses:

- linear interpolation for arm joints
- linear interpolation for gripper joints / commands
- linear interpolation for translation + quaternion slerp for poses

That is good enough for presentation replay, but it is not a more advanced:

- dynamics-aware smoother
- controller-aware reconstruction
- velocity / acceleration continuity optimizer

So the goal is “looks smoother”, not “perfectly reconstruct the original low-level control trajectory”.

## Recommended settings

For a natural-looking but not overly slow replay:

- `--interp_factor 4`
- `--fps 20`

For quick pipeline checking:

- `--interp_factor 2`
- `--max_input_records 20`

## Batch commands

There is currently no dedicated batch wrapper for the smooth replay path in the repo yet, so the most direct batch workflow is a shell `for` loop.

For your current output root:

- `/home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_single_close_only_viewer_visual07_collision03`

```bash

  [ -d "$d" ] || continue

    echo "[skip] missing plan_summary.json: $d"
```

## Relation to execution tuning

This replay tool is best used in the following workflow:

1. tune real execution parameters first
   - `joint_command_scene_steps`
   - `settle_steps`
   - `joint_target_wait_steps`
2. confirm real `reached / pos_err / rot_err`
3. then use `pose_debug.jsonl` for post-hoc presentation smoothing

Recommended principle:

- solve **real execution performance** in execution parameters
- solve **video appearance** with `replay_pose_debug_smooth.py`
