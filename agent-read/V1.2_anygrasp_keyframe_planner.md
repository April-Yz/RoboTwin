# V1.2 AnyGrasp Keyframe Planner

## What changed
- Added a user-facing reproduction document that explains the workflow without requiring code reading.
- Improved debug visualization for white-background scenes by changing arm markers from white to black.
- Added projected candidate labels to debug videos.

## New visualization semantics
- Green candidate actor + numeric label: raw candidate set for the active keyframe.
- Blue candidate actor + blue numeric label: left-hand ranked candidate set.
- Orange candidate actor + orange numeric label: right-hand ranked candidate set.
- Red candidate actor + red numeric label: final selected candidate.
- Left arm uses blue for top-ranked candidates.
- Right arm uses orange for top-ranked candidates.
- Selected candidates are red.
- Candidate IDs are shown as smaller plain numbers without background boxes or `L/R` prefixes.

## Why this change was needed
The previous left/right marker used white geometry, which was hard to see against the white background in head camera debug videos. Users also could not reliably map visualized candidates back to `candidate_idx` entries in AnyGrasp JSON outputs.

## Affected files
- `code_painting/plan_anygrasp_keyframes_r1.py`
- `code_painting/README_anygrasp_keyframe_planner.md`
- `code_painting/README_anygrasp_keyframe_planner_repro.md`

## Expected behavior
Running the planner in debug mode now produces videos where a human can identify:
- which raw AnyGrasp candidate corresponds to each visible gripper
- whether that candidate is being considered by the left or right hand ranking pass
- which candidate was finally selected
