# RoboTwin-lingbot V1.1 LingBot Evaluation Setup

## What Changed

- Added a dedicated RoboTwin worktree and matching conda environment for LingBot-VA evaluation.
- Updated the declared `huggingface_hub` dependency from `0.25.0` to `0.36.2`.
- Added a planner fallback path so evaluation can continue with `MPLib` when optional `curobo` import or build fails on this machine.
- Added result ignore rules for local LingBot smoke-test outputs.
- Added a CPU farthest-point fallback so missing `pytorch3d` does not abort RGB-based LingBot runs.
- Added `task_config/demo_clean_large_d435.yml` for the `Large_D435` camera layout used by the recollected `640x480` data pipeline.

## Why

LingBot-VA's RoboTwin evaluation instructions require a newer `huggingface_hub` version than the baseline RoboTwin setup in this repository declared.
The current machine uses a Blackwell GPU, while the available local CUDA 12.1 toolchain cannot compile `curobo` for `sm_120`, so evaluation needed a non-curobo fallback to stay runnable.

## Affected Areas

- `script/requirements.txt`
- LingBot-VA evaluation now targets this worktree instead of the shared RoboTwin checkout.
- `envs/robot/robot.py`
- `envs/robot/planner.py`
- `.gitignore`
- `envs/camera/camera.py`

## Usage Notes

- Activate `RoboTwin-lingbot` before running RoboTwin-side evaluation commands.
- Keep LingBot-VA server commands in the separate `lingbot-va` environment.
- If `curobo` fails to import or tries to JIT compile unsupported kernels, RoboTwin now falls back to `mplib_RRT` automatically instead of aborting at import time.
- A minimal smoke test succeeded with `click_bell` and `test_num=1`; generated videos and metrics remain local-only because `eval_result/` and `smoke-results/` are ignored.
- The separate `lingbot-va` action-only DSRL entry has now also completed one full `click_bell` online episode against this worktree. The runtime pipeline is functional even though that test episode did not solve the task.
