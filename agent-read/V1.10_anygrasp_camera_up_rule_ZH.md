# V1.10 AnyGrasp 相机朝上规则

## 这次改了什么

给 AnyGrasp candidate 加了一层 planner 侧姿态后处理：

- `--candidate_keep_camera_up 1`
- `--candidate_camera_top_axis z|y`

打开后，planner 会保留原本的抓取方向，只保守地处理夹爪绕前进轴的冗余自转。它会优先选择“顶部轴朝上的那一侧”，而不是把顶部轴强行精确对齐到世界 `+Z`。

## 为什么要加这个规则

这次用户观察到：

- keyframe 1 和 keyframe 22 的夹爪姿态，左手看起来差不多绕自己转了 `360` 度
- 但对抓取来说，这两种姿态其实等价

这说明当前姿态里有一维是冗余自由度：

- TCP 位置没变
- 前进/接近方向没变
- 夹爪开口含义没变
- 只是绕前进轴自转了

这对抓取本身通常不敏感，但会影响：

- 夹爪顶部朝哪里
- 夹爪上方安装的小相机朝哪里

用户希望最终行为是：

- 相机始终在上方
- 不允许夹爪绕前进轴随意自转

## 哪一轴是这个“360 度等价”的来源

当前 planner 和机器人链路都表明：

- gripper 的局部 `+X` 就是前进轴 / 接近轴

代码证据：

- [`code_painting/plan_anygrasp_keyframes_r1.py`](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py) 里的 `pose_with_offset_along_local_x(...)`
- [`envs/robot/robot.py`](/home/zaijia001/ssd/RoboTwin/envs/robot/robot.py) 里的 `_trans_from_gripper_to_endlink(...)`

所以真正冗余的那一维就是：

- 绕局部 `+X` 的 roll

## 新规则具体怎么做

对每个 AnyGrasp candidate：

1. 先像以前一样，把 camera-space 姿态转成 world-space
2. 如果开了 `--candidate_keep_camera_up 1`：
   - 保持局部 `+X` 不变
   - 比较两种姿态：
     - 原始 candidate 姿态
     - 绕局部 `+X` 再转 `180` 度后的姿态
   - 选择那个“顶部轴朝上分量更大”的版本

这样得到的效果是：

- 接近方向不变
- 不会把 roll 强行吸到某一个固定值
- 侧向抓取如果本来就没倒置，就尽量保持原样
- 只有在“顶部/相机朝下”时，才会翻到朝上的那一面

这个实现是故意保守的。它解决的是“上下颠倒”的冗余，而不是把所有候选都压到同一个精确 roll。

## 新参数

- `--candidate_keep_camera_up 1`
  - 打开“去掉绕前进轴自由滚转”的规则
- `--candidate_camera_top_axis z`
  - 把局部 `+Z` 当成夹爪顶部 / 相机朝上方向
- `--candidate_camera_top_axis y`
  - 如果后续确认相机安装方向其实对应局部 `+Y`，就改用这个

当前默认建议先用：

- `--candidate_camera_top_axis z`

## 结果里怎么检查

`plan_summary.json` 现在会额外保存：

- `candidate_keep_camera_up`
- `candidate_camera_top_axis`
- `top_axis_up_dot`

其中 `top_axis_up_dot` 表示“顶部轴”和世界 `+Z` 的点积：

- 接近 `1.0`：顶部朝上
- 接近 `0.0`：顶部水平
- 小于 `0`：顶部朝下

另外现在还会记录：

- `original_top_axis_up_dot`
- `camera_up_flip_applied`
- `forward_axis_change_deg`

正常情况下应该看到：

- `forward_axis_change_deg = 0.0`
- 只有在确实用了“翻到朝上那一侧”的版本时，`camera_up_flip_applied = 1`

## 你前面一次改动了什么，和这次成功有什么区别

最近和这条链路直接相关的改动可以分成两类。

真正改变行为的：

- `d877396`
  - 在 `action` 阶段回放物体轨迹
- `830f057`
  - 支持给 replay 物体替换 mesh，比如换成 `blue_cup.obj`

只用于 debug 的：

- `27a4bc9`
  - 导出 replay 侧的 pose debug
- `6a0974f`
  - 导出 planner 侧的 pose debug，并比较“导出 world pose 时的 head cam”和“当前 planner 的 head cam”

所以如果这次运行变正确了，直接影响行为的主要是：

- action 阶段的物体 replay
- mesh override

后两个 pose debug 提交本身主要是为了排查，不是为了改变 motion logic。

## 上一次为什么错了

上一次我的理解偏硬了。

我当时实现成了：

- 保持局部 `+X` 不变
- 再把顶部轴尽量精确对齐到世界上方

这个数学上没问题，但工程上约束太强。结果就是：

- 前进轴虽然没变
- 但视觉上的 roll 被改得太狠
- 某些本来侧向合理的抓取，看起来反而被“掰正”了

而你真正要的是：

- 不要让相机在下方
- 但也不要把夹爪固定到一个死的 roll

所以现在改成更保守的规则：

- 只解决“上下颠倒”
- 不去强行消灭所有 roll 差异
- 尽量保留原始侧向姿态
