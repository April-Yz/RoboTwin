# V1.15 URDF IK Cartesian Interp Execution Semantics

## Conclusion

After the latest fix, under `planner_backend=urdfik` and `urdfik_trajectory_mode=cartesian_interp_ik`:

- intermediate `ee/tcp pose` interpolation is generated
- IK is solved at each intermediate waypoint
- the resulting `joint_waypoints` are now actually executed segment by segment

So the current meaning of this path is now:

- `cartesian_interp_ik = solve IK along Cartesian waypoints and execute the resulting joint-waypoint sequence`

## Why this happens

### 1. The planning stage really does generate Cartesian waypoints

In `render_hand_retarget_r1_npz_urdfik.py`:

- `_plan_path_cartesian_interp_ik(...)`

it does the following:

1. reads the current TCP pose and target TCP pose
2. generates a TCP waypoint series with `_interpolate_tcp_pose_world_series(...)`
3. runs `_solve_ik_best_candidate(...)` on each waypoint
4. collects a list of `joint_waypoints`

So:

- `ee/tcp pose` interpolation does exist
- `joint_waypoints` also do exist

### 2. The execution stage now consumes the full `joint_waypoints`

The actual execution happens in:

- `execute_plans(...)`
- `_execute_single_ik_plan(...)`

The execution path now directly consumes:

- `plan["position"]`
- `plan["velocity"]`

For dual-arm execution:

- left and right waypoint sequences are interleaved by relative progress

For single-arm execution:

- the full `position` sequence is replayed in order

So:

- the joint waypoints produced from intermediate Cartesian IK are now part of the actual executed path

## What behavior this creates

### 1. `cartesian_interp_ik` still affects the IK result

Because the intermediate waypoints influence:

- the IK seed at each step
- which local basin the solver falls into
- which final `target_joints` are selected

So this feature is not “unused”.

### 2. It now directly affects the executed trajectory shape

Because the robot now actually executes:

- `joint_waypoints[0] -> joint_waypoints[1] -> ... -> joint_waypoints[n]`

This means:

- `cartesian_interp_ik` is no longer only an IK-guidance trick
- it now has a direct effect on the final executed motion

## Relation to the new automatic waypoint mode

The recently added:

- `--urdfik_cartesian_interp_steps -1`

only changes how the number of waypoints is chosen:

- fewer waypoints for small moves
- more waypoints for larger moves

It does not change the new semantics above:

- the waypoints are now used both for IK guidance and for execution

## Current code locations

Planning and waypoint generation:

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `_plan_path_cartesian_interp_ik(...)`
  - `_interpolate_tcp_pose_world_series(...)`
  - `_solve_ik_best_candidate(...)`

Actual execution:

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `execute_plans(...)`
  - `_execute_single_ik_plan(...)`

## Viewer vs exported head-camera video: why interpolation can look smooth in the viewer but like a jump in `head_cam_plan.mp4`

This workflow has two different clocks:

1. **physics / viewer clock**
2. **video recording clock**

They are related, but not identical.

### 1. `urdfik_cartesian_interp_steps` belongs to planning

In `render_hand_retarget_r1_npz_urdfik.py`:

- `_plan_path_cartesian_interp_ik(...)`

`--urdfik_cartesian_interp_steps` controls how many **TCP Cartesian waypoints** are generated between the current TCP pose and the target TCP pose.

So with:

- `--urdfik_trajectory_mode cartesian_interp_ik`
- `--urdfik_cartesian_interp_steps 20`

planning first creates about 20 TCP waypoints, then solves IK waypoint by waypoint, producing a multi-point joint trajectory.

This is a **planning density** parameter, not a post-plan settling parameter.

### 2. `execute_interp_steps` can densify the joint trajectory again

In `plan_anygrasp_keyframes_r1.py`:

- `interpolate_joint_trajectory(...)`
- `execute_single_arm_plan(...)`
- `execute_dual_arm_plan(...)`

The joint trajectory coming from the planner is passed through `interpolate_joint_trajectory(position, velocity, execute_interp_steps)` before execution.

So even if planning already produced 20 joint waypoints, the executor may still densify them further.

Important detail:

- if the planned trajectory already has many points, `execute_interp_steps` may only increase the count slightly
- if the planned trajectory has very few points, `execute_interp_steps` is the main source of visible smoothness

For example, with around 20 planned points and `--execute_interp_steps 24`, execution can still expand the commanded joint sequence beyond 20 frames.

### 3. `joint_command_scene_steps` controls how many physics steps are given to each commanded waypoint

In `execute_single_arm_plan(...)` and `execute_dual_arm_plan(...)`:

- one joint waypoint is commanded
- then `renderer.step_scene(steps=joint_command_scene_steps)` is run
- then one video frame is recorded with `record_frame(...)`

So the exported video records **one frame per commanded waypoint**, not one frame per physics step.

### 4. `joint_target_wait_steps` is a post-trajectory convergence wait

After all commanded waypoints are finished, the code calls:

- `settle_arms_to_targets(...)`

This function:

- checks the current-vs-target joint error
- if the error is still larger than `joint_target_wait_tol_rad`
- keeps stepping physics up to `joint_target_wait_steps`

This phase is very important:

- the viewer still renders during these extra `step_scene(steps=1)` calls
- but `head_cam_plan.mp4` does **not** record every one of these settle steps
- the video only records one final `plan_step=done` frame after the settle loop

So if the arm is still physically catching up during the settle loop, then:

- **viewer:** you see the robot continue to move smoothly
- **exported head video:** you mainly see the pre-settle frame and then the final settled frame
- visually this can look like a sudden jump / teleport in the saved video

This is the main reason the viewer can look smoother than the exported head-camera video.

### 5. What your current command implies

With the command shown in this debugging round:

- `--urdfik_cartesian_interp_steps 20`
- `--joint_target_wait_steps 70`
- `--viewer_frame_delay 0.02`

this means:

- planning generates a relatively dense Cartesian waypoint path
- execution commands those waypoints (possibly after another light interpolation pass)
- after the commanded path, the system can still spend up to 70 extra physics steps waiting for joint convergence
- the viewer shows those settle steps in real time
- the saved head-camera video does not serialize those 70 settle steps frame-by-frame

So a very common observed pattern is:

- the viewer looks like continuous motion
- the saved video looks like the robot arrives abruptly near the end of the stage

### 6. Parameters that influence this effect

Most relevant parameters are:

- `--urdfik_cartesian_interp_steps`
  - number of Cartesian TCP planning waypoints
- `--execute_interp_steps`
  - post-plan joint-trajectory densification before command replay
- `--joint_command_scene_steps`
  - physics steps allocated per commanded waypoint before one frame is recorded
- `--joint_target_wait_steps`
  - extra post-trajectory convergence steps after the replay loop
- `--joint_target_wait_tol_rad`
  - when the settle loop is allowed to stop early
- `--settle_steps`
  - one extra coarse pause before the final settle loop
- `--viewer_frame_delay`
  - only slows human viewing; it does not add saved video frames
- `--fps`
  - controls how quickly the recorded frames are played back, not how many settle-loop frames are captured

### 7. Practical interpretation

If you want the saved video to show more of the same gradual motion that the viewer shows, the first place to inspect is not only planner interpolation density, but also whether too much visible motion is happening inside the unrecorded final settle loop.

In other words:

- `urdfik_cartesian_interp_steps` explains how dense the **planned path** is
- `joint_target_wait_steps` explains how long the simulator keeps letting the arm **physically catch up after the path has already been commanded**

If the controller response is slow relative to `joint_command_scene_steps`, a lot of visible motion can migrate from the recorded command loop into the unrecorded settle loop.

## What to inspect first during future debugging

If the problem is:

- “IK is unstable”
- “the final joint solution goes in the wrong direction”

inspect first:

- Cartesian waypoint generation
- per-waypoint IK solutions
- how seeds are passed forward

If the problem is:

- “the motion still does not look Cartesian”
- “the execution path looks rough or wrong”

inspect first:

- whether waypoint count is too dense or too sparse
- whether the generated `joint_waypoints` themselves are noisy
- whether dual-arm interleaving is introducing timing artifacts

The old limitation of “execution ignores `joint_waypoints`” no longer applies after this fix.
