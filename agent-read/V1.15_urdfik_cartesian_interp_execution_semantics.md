# V1.15 URDF IK Cartesian Interp Execution Semantics

## Conclusion

Under the current `planner_backend=urdfik` and `urdfik_trajectory_mode=cartesian_interp_ik` path:

- intermediate `ee/tcp pose` interpolation is indeed generated
- IK is indeed solved at each intermediate waypoint
- but those intermediate waypoints are **not executed segment by segment**
- the actual execution stage is still mainly:
  - `current_joints -> target_joints`
  - followed by one joint-space linear interpolation

So the real meaning of this path is closer to:

- `cartesian_interp_ik = use Cartesian waypoints to help IK find a more stable final joint target`

rather than:

- `cartesian_interp_ik = execute the robot strictly along those Cartesian waypoints`

## Why this happens

### 1. The planning stage really does generate Cartesian waypoints

In `render_hand_retarget_r1_npz_urdfik.py`:

- `_plan_path_cartesian_interp_ik(...)`

it does the following:

1. reads the current TCP pose and target TCP pose
2. generates a TCP waypoint series with `_interpolate_tcp_pose_world_series(...)`
3. runs `_solve_ik_best_candidate(...)` on each waypoint
4. collects a list of `joint_waypoints`

So:

- `ee/tcp pose` interpolation does exist
- `joint_waypoints` also do exist

### 2. But the execution stage does not consume the full `joint_waypoints`

The actual execution happens in:

- `execute_plans(...)`

This function does not execute `plan["position"]` or the full `joint_waypoints` sequence segment by segment.

What it currently does is:

1. read `current_joints`
2. read `target_joints`
3. run one `np.linspace(...)` between those two endpoints
4. send that joint-space straight-line trajectory for execution

So:

- the intermediate Cartesian waypoints are not used as the actual executed path
- they only influence how the final `target_joints` are solved

## What behavior this creates

### 1. `cartesian_interp_ik` still affects the IK result

Because the intermediate waypoints influence:

- the IK seed at each step
- which local basin the solver falls into
- which final `target_joints` are selected

So this feature is not “unused”.

### 2. But its effect on the final executed trajectory shape is limited

Because the robot is not actually executing:

- `joint_waypoints[0] -> joint_waypoints[1] -> ... -> joint_waypoints[n]`

Instead, it only executes:

- `current_joints -> final_target_joints`

This is why the observed motion can:

- look unlike a true Cartesian path
- differ from the intermediate waypoint path you expected
- still resemble plain joint interpolation even with `cartesian_interp_ik` enabled

## Relation to the new automatic waypoint mode

The recently added:

- `--urdfik_cartesian_interp_steps -1`

only changes how the number of waypoints is chosen:

- fewer waypoints for small moves
- more waypoints for larger moves

It does not change the fundamental execution semantics above:

- the waypoints are still primarily used for IK guidance
- the current execution path is still not a strict segment-by-segment replay of those waypoints

## Current code locations

Planning and waypoint generation:

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `_plan_path_cartesian_interp_ik(...)`
  - `_interpolate_tcp_pose_world_series(...)`
  - `_solve_ik_best_candidate(...)`

Actual execution:

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `execute_plans(...)`
  - `_execute_single_ik_plan(...)`

## What to inspect first during future debugging

If the problem is:

- “IK is unstable”
- “the final joint solution goes in the wrong direction”

inspect first:

- Cartesian waypoint generation
- per-waypoint IK solutions
- how seeds are passed forward

If the problem is:

- “the motion does not look Cartesian”
- “the execution path looks rough or wrong”

inspect first:

- whether `execute_plans(...)` actually consumes the full `joint_waypoints`

That is the key limitation of the current execution semantics.
