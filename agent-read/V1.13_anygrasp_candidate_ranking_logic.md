- [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py)

## 1. Short version

There are two different ranking concepts in this pipeline:

1. `render_anygrasp_ranked_preview.py`
- It is no longer only a raw-score preview.
- Stage 1 filters candidates by left/right target object and prints before/after counts in the terminal.
- Stage 2 ranks the filtered candidates by orientation similarity only and exports one combined left-right image.
- Stage 3 ranks the same filtered candidates by a fused score, defaulting to `0.5 * anygrasp_score + 0.5 * orientation_score`, and exports another combined left-right image.

2. `plan_anygrasp_keyframes_r1.py`
- Performs the real planner-side ranking and final selection.
- Converts candidates into world poses, matches each one to its nearest visible object, applies optional object and distance constraints, then sorts by:
  - `rotation_distance_deg` ascending
  - `nearest_object_distance_m` ascending
  - `score` descending
- After that, it selects a two-keyframe combination on the same object with minimum total cost.

As a result, a candidate that is top-1 in the raw-score preview is not necessarily the one the planner will execute.

## 3. Current logic in `render_anygrasp_ranked_preview.py`

## 3.1 Purpose

This script is still a preview tool, but it now mirrors the early planner-style filtering chain more closely. Its role is to show:

- how target-object filtering changes the candidate pool
- how left/right orientation ranking looks after object filtering
- how a fused AnyGrasp-plus-orientation score reorders the same pool

## 3.2 Inputs and replay prerequisite

The script now needs:

- `grasps/grasp_<frame>.json`
- `hand_detections_<id>.npz`
- `replay_dir/multi_object_world_poses.npz`

Important prerequisite:

- `multi_object_world_poses.npz` must contain `head_camera_pose_world_wxyz`
- because the script transforms replay object world positions back into camera coordinates to reproduce nearest-object matching without launching the full RoboTwin renderer

## 3.3 Stage 1: object filtering

The relevant functions are:

- `load_replay_frame_data(...)`
- `nearest_object_name_in_camera(...)`
- `build_candidates_for_arm(...)`

For each frame:

1. load replay object world poses
2. load `head_camera_pose_world_wxyz`
3. transform each visible object from world coordinates back into camera coordinates
4. assign each grasp its nearest object
5. keep only `nearest_object == left_target_object` for the left arm
6. keep only `nearest_object == right_target_object` for the right arm

The script prints counts like:

```text
[frame 000027][object-filter] before_total=60 left_before=60 right_before=60 left_after=12 target=cup right_after=9 target=bottle
```

## 3.4 Stage 2: orientation-only ranking

After object filtering, the script computes:

```python
orientation_score = 1 - rotation_distance_deg / 180
```

and sorts by:

```python
(-orientation_score, rotation_distance_deg, -anygrasp_score)
```

It exports:

- `frame_<frame>_left_right_orientation_rank.png`

Each panel line includes:

- candidate index
- orientation score
- rotation distance in degrees
- matched object name

## 3.5 Stage 3: fused-score ranking

The same filtered candidate pool is also ranked by:

```python
fused_score = anygrasp_score_weight * anygrasp_score + orientation_score_weight * orientation_score
```

The default is:

```python
0.5 * anygrasp_score + 0.5 * orientation_score
```

and the sort key is:

```python
(-fused_score, -anygrasp_score, -orientation_score)
```

It exports:

- `frame_<frame>_left_right_fused_rank.png`

The panel explicitly shows:

```text
scoreA*0.50 + scoreB*0.50 = total
```

## 4. Real planner-side ranking

The real ranking happens in [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py), mainly through:

- `build_ranked_candidates_for_arm(...)`
- `select_keyframes_for_arm(...)`
- `choose_keyframes(...)`

### 4.1 Candidate world-pose construction

Each AnyGrasp candidate is first converted by `candidate_to_world_pose(...)`:

1. load camera-frame `rotation_matrix` and `translation`
2. apply orientation remap and post-rotation
3. convert camera pose to world pose through `renderer.camera_to_world_pose(...)`
4. optionally shift along local `+X` by `--candidate_target_local_x_offset_m`
5. optionally apply the camera-up roll disambiguation if `--candidate_keep_camera_up=1`

This step does not rank candidates yet, but it changes the pose used by planning and visualization.

### 4.2 Nearest-object assignment

`nearest_object_name(...)` computes the Euclidean distance from the candidate world position to every visible replay object and stores:

- `nearest_object`
- `nearest_object_distance_m`

This gives each candidate an object label and an object-distance metric.

### 4.3 Per-arm, per-keyframe ranking

`build_ranked_candidates_for_arm(...)` does the planner-side ranking.

For each keyframe:

1. choose the arm reference rotation
- current hand frame if valid
- otherwise nearest valid hand frame

2. build candidate records
- raw AnyGrasp score
- world pose
- nearest object and object distance
- `rotation_distance_deg` from the arm reference rotation

3. apply optional object constraint
- if `--enforce_target_object_constraint=1`
- left arm keeps only `nearest_object == left_target_object`
- right arm keeps only `nearest_object == right_target_object`

4. apply optional distance constraint
- if `--enforce_candidate_distance_constraint=1`
- keep only `nearest_object_distance_m <= candidate_object_max_distance_m`

5. sort candidates

The sort key is:

```python
all_candidates.sort(key=lambda cand: (cand.rotation_distance_deg, cand.nearest_object_distance_m, -cand.score))
```

and the unfiltered `raw_candidates` list is sorted with the same key.

Important naming detail:

- `raw_candidates`
  - means the full list before object/distance filtering
  - not a raw-score-ranked list
- `all_candidates`
  - means the filtered list actually used for planner-side ranking

So the real planner preference is:

1. smaller hand-orientation mismatch
2. smaller distance to the nearest object
3. higher AnyGrasp score only as a tie-breaker

That is fundamentally different from the score-only preview script.

## 5. How the final two-keyframe pair is chosen

Per-frame ranking is not yet the final answer. The final pair is selected in `select_keyframes_for_arm(...)`.

### 5.1 Manual override has highest priority

`--manual_candidate FRAME ARM CANDIDATE_IDX` is parsed by `parse_manual_candidate_overrides(...)`.

`reorder_candidates_with_manual_choice(...)` moves the specified candidate to the front of the list.

If one arm has manual candidates for both keyframes:

- the planner uses those two candidates directly
- automatic pair search is skipped

If only some frames are manually specified:

- list order is adjusted
- automatic selection still runs

### 5.2 Automatic selection requires the same object across both keyframes

In automatic mode, the planner:

1. iterates over each first-keyframe candidate
2. takes its `nearest_object`
3. filters second-keyframe candidates to the same `nearest_object`

So the two selected keyframes must correspond to the same object.

### 5.3 Automatic pair cost

The pair cost is:

```python
total_cost = (
    first_candidate.rotation_distance_deg
    + best_second.rotation_distance_deg
    + 10.0 * (first_candidate.nearest_object_distance_m + best_second.nearest_object_distance_m)
)
```

This means:

- both keyframes prefer small hand-rotation error
- both keyframes prefer candidates closer to the matched object
- object distance is explicitly amplified by `10.0`
- raw AnyGrasp score is not directly used at this stage anymore

Why `best_second = second_candidates[0]` is enough:

- second-keyframe candidates have already been sorted by:
  - rotation distance
  - object distance
  - negative score
- so the first candidate for that object is already the best second-frame choice under that ordering

## 6. How the executing arm is chosen

`choose_keyframes(...)` handles left-vs-right selection.

- with `--arm left/right`, only that arm is evaluated
- with `--arm auto`, both arms are evaluated independently

The final arm comparison uses:

```python
total_cost = sum(item.candidate.rotation_distance_deg for item in selection.selected_keyframes)
```

So the left/right competition is based on the sum of the selected keyframes' rotation distances.

The object-distance term is not re-added here, because each arm's internal pair search has already filtered and ranked candidates using object distance.

## 7. Why preview rank and final selection can disagree

The most common reasons are:

1. the updated preview script still stops at single-frame object filtering plus two single-frame ranking views
2. the planner continues with its own formal ranking key of rotation error, object distance, then score
3. the planner performs cross-keyframe same-object pairing
4. the planner may choose between left and right arms in `--arm auto` mode

So a raw-score top-1 candidate can still be rejected during planner selection.

## 8. Commonly confused data structures

### 8.1 `ranked_candidates_per_frame`

- filtered planner-side candidates
- ordered in the real automatic-selection order
- may be manually reordered by `--manual_candidate`

### 8.2 `all_candidates_per_frame`

- the unfiltered full candidate set
- still sorted by planner ranking key, not raw score
- mainly used for debugging and manual inspection

### 8.3 `common_candidates_per_frame`

Built in `build_display_candidates_per_frame(...)`.

If `debug_show_all_candidates=1`, it takes `all_candidates_per_frame`, then re-sorts that display list by raw `score` descending before showing green common candidates.

So the debug view can simultaneously show two ranking semantics:

- green common candidates: score-oriented display order
- blue/orange arm-specific candidates: planner-ranked order

## 9. Difference between the two rank-preview tools

The names are similar, but they do different things:

1. [render_anygrasp_ranked_preview.py](/home/zaijia001/ssd/RoboTwin/code_painting/render_anygrasp_ranked_preview.py)
- no full RoboTwin execution loop
- draws directly on AnyGrasp images or replay RGB frames
- now exports:
  - object-filter terminal statistics
  - orientation-ranked combined images
  - fused-score-ranked combined images

2. `export_rank_preview_images(...)` inside [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py)
- uses `ranked_candidates_per_frame`
- exports rank-1 to rank-N images from the planner-side ordering
- is much closer to what the planner may actually execute

## 10. One-sentence summary

In one sentence:

- `render_anygrasp_ranked_preview.py` now shows target-object filtering, then orientation ranking, then fused AnyGrasp-plus-orientation ranking
- `plan_anygrasp_keyframes_r1.py` still goes further, using planner-side object distance, same-object two-keyframe pairing, and final arm selection
