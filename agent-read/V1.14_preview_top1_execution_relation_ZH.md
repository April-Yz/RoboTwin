# V1.14 Preview Top1 执行关系说明

这份说明专门记录当前基于 preview `summary.json` 的 AnyGrasp planner 执行逻辑。

相关代码：
- `code_painting/plan_anygrasp_keyframes_r1.py`
- `code_painting/plan_anygrasp_keyframes_r1_batch.py`

## 1. 候选来源

这个模式下，planner 在关键帧选择阶段不会重新扫描所有 AnyGrasp 候选。

它会直接读取 preview 输出 JSON：
- `render_anygrasp_ranked_preview.py` 产出的 `summary.json`
- 每只手对应的 top 列表：
  - `top_candidates.left_orientation`
  - `top_candidates.right_orientation`
  - 如果显式指定，也可以读 fused 列表

当参数是：
- `--reuse_preview_candidate_group orientation`
- `--reuse_preview_top_rank 1`

planner 实际读取的是：
- `top_candidates.left_orientation[0]`
- `top_candidates.right_orientation[0]`

也就是每帧每只手的 top1。

## 2. 关键帧规则

当前 preview-top1 复用路径支持两种关键帧模式：

- `--reuse_preview_frame_mode legacy_1_max22rel`
- `--reuse_preview_frame_mode annotated_json_keyframes`

### 2.1 旧模式：`legacy_1_max22rel`

如果命令里使用：
- `--keyframes 1 22`
- `--candidate_selection_relative_frame -10`

那么关键帧解析以 preview 的 `resolved_frames` 为准，最终执行帧是：
- 关键帧1 = `1` 对应解析后的帧
- 关键帧2 = `max(resolved(22), resolved(-10))`

对于当前这组 `fi60` preview：
- `resolved(-10) = 28`
- 所以最终两帧是 `1` 和 `28`

### 2.2 新模式：`annotated_json_keyframes`

这个模式用于：

- `/home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_direct_preview_keyframes_batch`

也就是 preview 已经按：
- `frame 0`
- 手工标注关键帧 1
- 手工标注关键帧 2

导出了三帧结果。

在这个模式下：
- planner 会读取 preview `summary.json` 里的：
  - `frame_selection.mode`
  - `frame_selection.annotated_keyframes`
- `frame 0` 只作为 preview 的上下文帧保留
- 真正执行的两帧是：
  - 第一个标注关键帧，作为 `grasp`
  - 第二个标注关键帧，作为 `action`

也就是说，原来的：
- `frame 1`
- `max(22, resolved(-10))`

会被替换为：
- `annotated_keyframes[0]`
- `annotated_keyframes[1]`

## 3. 阶段顺序

当前执行顺序是：
1. 先到 `pregrasp`
2. 再从 `pregrasp` 到关键帧1抓取位姿
3. 闭合夹爪
4. 再移动到关键帧2动作位姿

这里的“关键帧1”指的是当前执行模式下的第一个执行关键帧：

- 在 `legacy_1_max22rel` 里，它通常是 `1`
- 在 `annotated_json_keyframes` 里，它是 `annotated_keyframes[0]`

`pregrasp` 不是这个关键帧本身。

它是用关键帧1抓取位姿，沿夹爪局部前进轴反向回退得到的：
- 偏移量 = `--approach_offset_m`
- 当前默认值 = `0.08` 米

因此当前行为实际就是：
- 先把 EE/TCP 平滑移动到 `pregrasp`
- 再从 `pregrasp` 短距离推进到关键帧1抓取位姿
- 闭合夹爪
- 再把 EE/TCP 平滑移动到关键帧2

对于推荐配置：
- `--planner_backend urdfik`
- `--urdfik_trajectory_mode cartesian_interp_ik`

平滑方式是先在 EE/TCP 笛卡尔空间插值，再逐路点做 IK。

## 4. 物体与夹爪的关系

在候选选择阶段，夹爪不会自动和物体建立“抓住了”的关系。

选择阶段：
- 夹爪位姿来自 AnyGrasp preview 的 top1
- 物体名称主要用于筛选和标注
- 这时还没有物理夹持或刚性附着关系

闭合前执行阶段：
- 在 `pregrasp` 和 `grasp` 阶段，物体保持在 replay 的关键帧1世界系位姿
- 夹爪独立朝着选中的 AnyGrasp 位姿移动

闭合后执行阶段：
- 如果 `--replay_objects_during_action 0`：
  - 选中的物体会在 `action` 阶段刚性附着到当前执行臂的 TCP
  - 没被选中的物体保持在固定 replay 位姿
- 如果 `--replay_objects_during_action 1`：
  - 所有物体按 `multi_object_world_poses.npz` 里的轨迹播放
  - 不会附着到 TCP

单臂默认模式：
- 只有一只手执行
- 只有这只手选中的物体会在 `action` 阶段有机会附着到 TCP

双臂同步模式：
- 如果双臂都执行，且关闭 object replay
- 那么左右手各自选中的物体会在闭合后分别附着到对应 TCP

## 4.1 为什么标注关键帧模式更容易暴露“没抓到就进入下一阶段”

之前这里存在一个全局流程问题，不是 `annotated_json_keyframes` 独有的问题：

- `execute_stage_until_reached()` 会正确返回 `reached=False`
- 但主流程以前没有根据这个结果阻断后续阶段
- 所以即使 `pregrasp` 或 `grasp` 没真正到位，也可能继续：
  - `close_gripper`
  - `action`

之所以你在新模式里更容易看到这个问题，原因是：

- `legacy_1_max22rel` 常用的第 1 帧抓取位姿通常更保守、更接近初始化状态
- `annotated_json_keyframes` 往往是你手工挑出来的动作关键帧，抓取位姿可能更激进、更靠近物体或姿态更极端
- 因此新模式更容易出现：
  - IK 可规划但末端误差仍超阈值
  - 或根本没达到 `reach_pos_tol_m / reach_rot_tol_deg`

所以表面上看像是“新模式有问题”，本质上是：
- 新模式更容易挑到难姿态
- 历史默认配置下每个阶段只会尝试 `3` 次
- 而且上一个临时修补版本又把“未到位就直接中止后续阶段”加得过于激进

当前行为已经改成：
- `try=3` 的来源是 `--max_stage_replans 3`
- 现在默认打开 `--replan_until_reached 1`
- 现在默认 `--replan_until_reached_max_attempts 0`
- 当 `replan_until_reached_max_attempts <= 0` 时，表示不设上限
- 所以若 `reach=False`，planner 会持续从当前位置重新规划并重复执行，直到真正满足 `reach_pos_tol_m / reach_rot_tol_deg`
- 同时也取消了“pregrasp 没到位就直接结束整个后续流程”的中止逻辑

## 5. 有头 Viewer 模式

这条 preview-top1 流程也支持有头界面运行。

常用参数：
- `--enable_viewer 1`
- `--viewer_frame_delay 0.02`
- `--viewer_wait_at_end 1`

这些参数不会改变选择逻辑，只是把同一套执行流程显示到交互窗口里。

## 5.1 纯净输出模式

新增参数：
- `--pure_scene_output 1`

作用：
- `head_cam_plan.mp4`
- `third_cam_plan.mp4`

会变成纯场景输出：
- 没有左上角文字
- 没有彩色候选夹爪
- 没有目标坐标轴

也就是只保留机器人和场景本身。

这个参数不会取消调试产物本身：
- `debug_selection_preview.mp4`
- `debug_execution_preview.mp4`

这些调试视频仍然可以保留原有可视化，方便排查问题。

推荐组合：
- `--pure_scene_output 1`
- `--save_debug_preview 1`
- `--save_debug_execution_preview 1`

这样你会同时得到：
- 纯净主视频：只看真实执行场景
- debug 视频：保留坐标系、抓取可视化、阶段信息

## 5.2 `legacy_1_max22rel` 的执行流程

当你使用：
- `--reuse_preview_frame_mode legacy_1_max22rel`
- `--keyframes 1 22`
- `--candidate_selection_relative_frame -10`

执行流程是：
- 先从 preview `summary.json` 里读取 `resolved_frames`
- 解析第二关键帧为 `max(22, resolved(-10))`
- 若 `--reuse_preview_top_rank 1`，则直接读取对应帧的 top1 候选
- 用关键帧1抓取位姿计算 `pregrasp = grasp 沿夹爪局部前进轴回退 8cm`
- 阶段顺序是 `init -> pregrasp -> grasp -> close_gripper -> keyframe_2`
- 若 backend 是 `urdfik` 且 `--urdfik_trajectory_mode cartesian_interp_ik`，则每个阶段会先插值出 `--urdfik_cartesian_interp_steps` 个中间 EE/TCP 目标，再逐段执行 IK
- 每次阶段执行完成后都会检测：
  - `pos_err_m`
  - `rot_err_deg`
- 历史行为是最多试 `3` 次
- 当前默认行为是：若 `reach=False`，继续从当前位置重规划并重复执行，直到到位

## 5.3 与 `annotated_json_keyframes` 的对比

现在两种模式的主要区别只剩关键帧来源：
- `legacy_1_max22rel`：`1 + max(22, resolved(-10))`
- `annotated_json_keyframes`：`frame_selection.annotated_keyframes[:2]`

其余执行逻辑一致：
- 都直接从 preview `summary.json` 读取 top1
- 都先算 `pregrasp`
- 都走 `init -> pregrasp -> grasp -> close_gripper -> keyframe_2`
- 都在阶段结束后检查是否到位
- 都会在 `reach=False` 时继续重规划直到到位

## 7. 关键参数怎么调

路径相关：
- `anygrasp_batch_results`
  - 原始 AnyGrasp 结果根目录
- `replay_m_obj_pose_d_pour_blue_norobot`
  - replay 物体轨迹根目录
- `hand_vis`
  - `hand_detections_<id>.npz` 所在目录
- `anygrasp_plan_keyframes_from_preview_fi60`
  - 本次 planner 输出目录
- `--reuse_preview_summary_root`
  - preview 批处理结果根目录，planner 会自动到每个子目录下找 `summary.json`

候选来源相关：
- `--reuse_preview_frame_mode legacy_1_max22rel|annotated_json_keyframes`
  - `legacy_1_max22rel`：沿用旧的 `1 + max(22, resolved(-10))`
  - `annotated_json_keyframes`：直接用 preview `frame_selection.annotated_keyframes` 里的前两帧
- `--reuse_preview_candidate_group orientation|fused`
  - `orientation`：按 hand orientation 排序后的 top 列表取候选
  - `fused`：按 AnyGrasp 分数和 orientation 分数融合后的列表取候选
- `--reuse_preview_top_rank 1`
  - 取 top1
  - 改成 `2` 就是取 top2

关键帧相关：
- `--keyframes 1 22`
  - 基础两帧
- `--candidate_selection_relative_frame -10`
  - 再引入相对帧
  - 最终第二帧是 `max(22, resolved(-10))`
- 上面这两项只用于 `legacy_1_max22rel`
- 在 `annotated_json_keyframes` 下：
  - 不再使用 `1 / 22 / -10`
  - 改为直接使用 `frame_selection.annotated_keyframes[:2]`
- `--candidate_selection_mode top_score_auto`
  - 在 preview-top1 这条路径里，它主要用来保持“auto 选手时按分数优先”的语义一致

运动方式相关：
- `--planner_backend urdfik`
  - 当前推荐
- `--urdfik_trajectory_mode cartesian_interp_ik`
  - 先走 EE/TCP 笛卡尔平滑，再逐点 IK
- `--urdfik_cartesian_interp_steps 30`
  - 越大越平滑，速度越慢
- `--approach_offset_m 0.08`
  - 关键帧1前退 8cm 作为 `pregrasp`

执行模式相关：
- `--arm auto`
  - 左右手都先读各自 top1，再决定执行哪一只手
- `--arm left|right`
  - 强制只执行某一只手
- `--execute_both_arms 1`
  - 双臂同步执行
- `--execute_both_arms 0`
  - 单臂执行，当前更稳

物体关系相关：
- `--replay_objects_during_action 0`
  - 闭合后，选中物体附着到 TCP
- `--replay_objects_during_action 1`
  - 物体继续按 replay 轨迹走，不附着 TCP

调试和显示相关：
- `--pure_scene_output 1`
  - `head_cam_plan.mp4` / `third_cam_plan.mp4` 输出纯场景
  - 不显示 overlay、候选夹爪、目标轴
- `--save_debug_preview 1`
  - 保存关键帧选择预览视频
- `--save_debug_execution_preview 1`
  - 保存慢速执行预览视频
- `--enable_viewer 1`
  - 打开有头界面
- `--viewer_frame_delay 0.02`
  - viewer 播放速度
- `--viewer_wait_at_end 1`
  - 结束后不立刻退出
