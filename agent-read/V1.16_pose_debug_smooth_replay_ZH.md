# V1.16 基于 pose_debug.jsonl 的平滑重放

## 目的

当 `plan_anygrasp_keyframes_r1.py` 使用：

- `--pure_scene_output 1`

时，主视频会保持纯净输出，同时自动写出：

- `pose_debug.jsonl`

这个文件会记录每一帧的：

- 左右臂关节角
- gripper 命令与 gripper joint qpos
- 头相机 / TCP / EE pose
- 当前执行物体 actor pose

基于这些记录，现在新增了一个**后处理平滑重放**脚本：

- `code_painting/replay_pose_debug_smooth.py`
- 启动包装：`code_painting/run_replay_pose_debug_smooth.sh`

它的目标是：

- 不改原始规划与执行结果
- 只把已经记录下来的离散执行帧做更密的插值重放
- 导出一个视觉上更平滑的 replay 视频

## 核心语义

这条新链路是：

1. 读取 `pose_debug.jsonl`
2. 在相邻记录帧之间做插值：
   - arm joint qpos：线性插值
   - gripper joint qpos / gripper command：线性插值
   - object actor pose：位置线性插值，旋转用 quaternion slerp
3. 用新的渲染器把插值后的状态重新渲染成视频

所以它属于：

- **post-hoc visual replay smoothing**

而不是：

- 在线控制补偿
- 重新规划
- 提高原始执行到位精度

更直白地说，这个 replay 的依据就是原始 run 里保存下来的状态记录。当前脚本直接使用这些字段：

- `current_left_arm_qpos_rad`
- `current_right_arm_qpos_rad`
- `current_left_gripper_joint_qpos_rad`
- `current_right_gripper_joint_qpos_rad`
- `current_left_gripper_command`
- `current_right_gripper_command`
- `object_actor_poses.*.actor_pose_world_wxyz`

然后在这些已记录状态之间补插值，再重新渲染。

所以答案是：

- **是的，它就是根据原结果中保存下来的关节位置、gripper 状态、物体 pose 等信息去直接做平滑和 replay。**
- 它不是重新跑 planner，也不是重新做 IK。

## 记录内容与视频帧的对齐关系

`pose_debug.jsonl` 是在 `plan_anygrasp_keyframes_r1.py` 的 `record_frame(...)` 里写的。

这意味着：

- 每调用一次 `record_frame(...)`
  - 就会写一帧主视频
  - 也会追加一条 `pose_debug.jsonl` 记录
- 因此它和**被实际录下来的视频帧**是按记录序号对齐的
- 同一条记录里会同时保存：
  - `current_left/right_arm_qpos_rad`
  - `current_left/right_gripper_joint_qpos_rad`
  - `current_left/right_gripper_command`
  - `current_left/right_tcp_pose_world_wxyz`
  - `current_left/right_ee_pose_world_wxyz`
  - `current_head_camera_pose_world_wxyz`
  - `object_actor_poses`

但要特别注意：

- 它不是“每个 physics step 都存一次”
- 它只是在代码显式调用 `record_frame(...)` 时才存

所以对齐关系是：

- **和导出视频帧对齐：是的**
- **和底层所有 physics 子步逐步对齐：不是**

这也是为什么原始执行里某些 motion 看起来仍然比较稀疏：

- command loop 里每个 waypoint 通常只录一帧
- settle / wait 阶段也不是每个 physics step 都录

## 重要限制

这个脚本改善的是：

- 视频观感
- 演示连续性

它**不会改变**：

- 原始 run 的 reach 结果
- 原始 run 的 `plan_summary.json` 中误差
- 原始 run 的真实控制收敛行为

一句话说：

- 它让“回放视频”更平滑
- 但不会让“当时那次执行”变得更准确

## 推荐使用场景

适合：

- pure 模式下已经跑完一遍
- 想基于那次结果导出更平滑的展示视频
- 不想重新跑 AnyGrasp / 规划 / 执行

不适合：

- 拿它判断控制器是否真的跟上
- 拿它代替 reach / attempt / pos_err / rot_err 的分析

## 关键参数

- `--plan_summary_json`
  - 优先推荐。自动推断 `pose_debug.jsonl` 和 `replay_dir`
- `--pose_debug_jsonl`
  - 直接指定重放源
- `--replay_dir`
  - 用来加载 replay 里的物体 mesh
- `--output_path`
  - 输出视频路径
- `--interp_factor`
  - 每个相邻记录帧之间插多少个子帧
  - `1` 表示不额外插值
  - 推荐从 `3` 或 `4` 开始
- `--fps`
  - 输出视频帧率
- `--overlay_text`
  - 是否在平滑重放视频上写状态字
- `--max_input_records`
  - 只重放前 N 条记录，适合快速 smoke test

## 平滑原理：它到底在平滑什么

平滑脚本不是去改 planner 的 20 个 Cartesian waypoint，也不是重跑控制器。

它做的是：

- 以 `pose_debug.jsonl` 中**已经被记录下来的离散帧**为关键帧
- 在相邻两条记录之间插入更多中间子帧
- 然后重新渲染出一条更密的视频

也就是说它是：

- **帧间平滑 / record-to-record interpolation**

不是：

- 直接把底层 controller 的每个物理子步补回来

因此如果你觉得“原来那 20 个插值的平滑效果有限”，通常可能有两类原因：

### 情况 A：真实变化本来就没有被足够密地记录下来

这是最常见的。

例如原始链路里：

- 每个 command waypoint 只录 1 帧
- 但每个 waypoint 内部实际只给了少量 physics step
- 后面还有 `settle_steps` / `joint_target_wait_steps` 这样的收敛阶段
- 这些阶段并不是每个 physics 子步都录主视频

那么 `pose_debug.jsonl` 里天然就只有比较稀的关键状态。

这种情况下，后处理平滑只能：

- 在这些稀疏关键状态之间做更顺的过渡

但不能恢复：

- 当时没有被记录下来的真实中间运动细节

### 情况 B：当前平滑方式本身比较简单

当前平滑策略是：

- arm joints：线性插值
- gripper joints / command：线性插值
- pose：位置线性插值 + quaternion slerp

这已经足够做展示视频，但它不是更高级的：

- dynamics-aware smoothing
- controller-aware reconstruction
- velocity / acceleration continuity optimization

所以它的目标是“更顺眼”，不是“还原真实控制轨迹的全部细节”。

## 推荐参数经验

如果你想得到更自然但不过分慢的回放：

- `--interp_factor 4`
- `--fps 20`

如果你只是快速检查 replay 链路是否正常：

- `--interp_factor 2`
- `--max_input_records 20`

## 批处理命令

当前仓库里还没有单独现成的 batch wrapper，所以最直接的批处理方式是 shell `for` 循环。

针对你现在这条输出目录：

- `/home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_single_close_only_viewer_visual07_collision03`

可以直接用：

```bash
for d in /home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_single_close_only_viewer_visual07_collision03/d_pour_blue_*; do
  [ -d "$d" ] || continue
  if [ ! -f "$d/plan_summary.json" ]; then
    echo "[skip] missing plan_summary.json: $d"
    continue
  fi
  out="$d/head_cam_plan_smooth.mp4"
  echo "[smooth] $d -> $out"
  bash /home/zaijia001/ssd/RoboTwin/code_painting/run_replay_pose_debug_smooth.sh \
    --plan_summary_json "$d/plan_summary.json" \
    --output_path "$out" \
    --interp_factor 4 \
    --fps 20 \
    --overlay_text 0 \
    --base_occluder_enable 0
 done
```

如果你只想批量处理部分 id，例如 `0 1 2 3`：


## 和执行调参的关系

这条新 replay 脚本适合放在下面这个工作流里：

1. 先调真实执行参数
   - `joint_command_scene_steps`
   - `settle_steps`
   - `joint_target_wait_steps`
2. 确认真实 `reached / pos_err / rot_err`
3. 再用 `pose_debug.jsonl` 做 post-hoc 平滑展示

推荐原则：

- **真实性能问题**优先在执行参数里解决
- **视频观感问题**再交给 `replay_pose_debug_smooth.py`
