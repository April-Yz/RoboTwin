# V1.16 基于 pose_debug.jsonl 的平滑重放

## 目的

当 `plan_anygrasp_keyframes_r1.py` 使用：

- `--pure_scene_output 1`

时，主视频会保持纯净输出，同时自动写出：

- `pose_debug.jsonl`

这个文件会记录每一帧的：

- 左右臂关节角
- gripper 命令与 gripper joint qpos
- 头相机 / TCP / EE pose
- 当前执行物体 actor pose

基于这些记录，现在新增了一个**后处理平滑重放**脚本：

- `code_painting/replay_pose_debug_smooth.py`
- 启动包装：`code_painting/run_replay_pose_debug_smooth.sh`

它的目标是：

- 不改原始规划与执行结果
- 只把已经记录下来的离散执行帧做更密的插值重放
- 导出一个视觉上更平滑的 replay 视频

## 核心语义

这条新链路是：

1. 读取 `pose_debug.jsonl`
2. 在相邻记录帧之间做插值：
   - arm joint qpos：线性插值
   - gripper joint qpos / gripper command：线性插值
   - object actor pose：位置线性插值，旋转用 quaternion slerp
3. 用新的渲染器把插值后的状态重新渲染成视频

所以它属于：

- **post-hoc visual replay smoothing**

而不是：

- 在线控制补偿
- 重新规划
- 提高原始执行到位精度

## 重要限制

这个脚本改善的是：

- 视频观感
- 演示连续性

它**不会改变**：

- 原始 run 的 reach 结果
- 原始 run 的 `plan_summary.json` 中误差
- 原始 run 的真实控制收敛行为

一句话说：

- 它让“回放视频”更平滑
- 但不会让“当时那次执行”变得更准确

## 推荐使用场景

适合：

- pure 模式下已经跑完一遍
- 想基于那次结果导出更平滑的展示视频
- 不想重新跑 AnyGrasp / 规划 / 执行

不适合：

- 拿它判断控制器是否真的跟上
- 拿它代替 reach / attempt / pos_err / rot_err 的分析

## 最常用命令

### 1. 直接从 `plan_summary.json` 推断输入

```bash
bash /home/zaijia001/ssd/RoboTwin/code_painting/run_replay_pose_debug_smooth.sh \
  --plan_summary_json /path/to/d_pour_blue_35/plan_summary.json \
  --output_path /path/to/d_pour_blue_35/head_cam_plan_smooth.mp4 \
  --interp_factor 4 \
  --fps 20 \
  --overlay_text 0 \
  --base_occluder_enable 1 \
  --base_occluder_local_pos 0.0 0.0 0.4 \
  --base_occluder_half_size 0.45 0.45 0.02 \
  --base_occluder_color 1.0 1.0 1.0
```

含义：

- 自动从 `plan_summary.json` 读取：
  - `pose_debug.jsonl`
  - `replay_dir`
- 每两个记录帧之间补 4 倍密度的重放子帧
- 导出新的平滑视频

### 2. 直接指定 `pose_debug.jsonl`

```bash
bash /home/zaijia001/ssd/RoboTwin/code_painting/run_replay_pose_debug_smooth.sh \
  --pose_debug_jsonl /path/to/d_pour_blue_35/pose_debug.jsonl \
  --replay_dir /path/to/replay_m_obj_pose_d_pour_blue_norobot/d_pour_blue_35 \
  --output_path /path/to/d_pour_blue_35/head_cam_plan_smooth.mp4 \
  --interp_factor 4 \
  --fps 20
```

## 关键参数

- `--plan_summary_json`
  - 优先推荐。自动推断 `pose_debug.jsonl` 和 `replay_dir`
- `--pose_debug_jsonl`
  - 直接指定重放源
- `--replay_dir`
  - 用来加载 replay 里的物体 mesh
- `--output_path`
  - 输出视频路径
- `--interp_factor`
  - 每个相邻记录帧之间插多少个子帧
  - `1` 表示不额外插值
  - 推荐从 `3` 或 `4` 开始
- `--fps`
  - 输出视频帧率
- `--overlay_text`
  - 是否在平滑重放视频上写状态字
- `--max_input_records`
  - 只重放前 N 条记录，适合快速 smoke test

## 推荐参数经验

如果你想得到更自然但不过分慢的回放：

- `--interp_factor 4`
- `--fps 20`

如果你只是快速检查 replay 链路是否正常：

- `--interp_factor 2`
- `--max_input_records 20`

## 代码位置

- 主脚本：
  - `code_painting/replay_pose_debug_smooth.py`
- shell 包装：
  - `code_painting/run_replay_pose_debug_smooth.sh`
- pose_debug 生成位置：
  - `code_painting/plan_anygrasp_keyframes_r1.py`
  - `record_frame(...)`

## 和执行调参的关系

这条新 replay 脚本适合放在下面这个工作流里：

1. 先调真实执行参数
   - `joint_command_scene_steps`
   - `settle_steps`
   - `joint_target_wait_steps`
2. 确认真实 `reached / pos_err / rot_err`
3. 再用 `pose_debug.jsonl` 做 post-hoc 平滑展示

推荐原则：

- **真实性能问题**优先在执行参数里解决
- **视频观感问题**再交给 `replay_pose_debug_smooth.py`
