# V1.7 AnyGrasp Pipeline

## Scope

This document covers the RoboTwin-side workflow after AnyGrasp results have already been produced on another server.

Boundary:
- This repo does not run AnyGrasp inference as the normal pipeline entrypoint.
- It consumes AnyGrasp outputs and turns them into replay, debug, and execution results.

## Inputs

Each video id needs three inputs:

1. AnyGrasp results
   - `grasps/grasp_<frame>.json`
2. Replay result
   - `multi_object_world_poses.npz`
3. Hand result
   - `hand_detections_<id>.npz`

Typical roots:

- AnyGrasp:
  - `/home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_batch_results`
- Replay:
  - `/home/zaijia001/ssd/RoboTwin/code_painting/replay_m_obj_pose_d_pour_blue`
- Hand:
  - `/home/zaijia001/ssd/data/R1/gt_depth_vis/d_pour_blue/hand_vis`

## Main Scripts

- Batch runner: [run_plan_anygrasp_keyframes_r1_batch.sh](/home/zaijia001/ssd/RoboTwin/code_painting/run_plan_anygrasp_keyframes_r1_batch.sh)
- Batch planner: [plan_anygrasp_keyframes_r1_batch.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1_batch.py)

## Logic

1. Load grasp candidates for two keyframes.
2. Load hand orientations from `hand_detections_<id>.npz`.
3. Rank left/right candidates.
4. Optionally apply object-name and distance constraints.
5. Optionally accept manual candidate overrides.
6. Plan `pregrasp -> grasp -> action`.
7. Save videos, summaries, and rank preview images.

## Outputs

Per-video:

- `debug_selection_preview.mp4`
- `debug_execution_preview.mp4`
- `head_cam_plan.mp4`
- `rank_previews/keyframe_<frame>_rank_<k>.png`
- `plan_summary.json`

Batch:

- `batch_plan_summary.json`

### Where object coordinates come from

All object coordinates come from `multi_object_world_poses.npz`, and they are already in the world frame. There is no extra camera-to-world conversion for object replay data.

There are two loading modes:

1. Static state at a keyframe
   - Used for candidate filtering, nearest-object matching, and initial visualization
   - Code: `load_object_states(...)`
   - It finds the requested `source_frame` in `selected_source_frame_indices`
   - Then loads that frame from `cup__pose_world_wxyz` / `cup__pose_world_matrix` / `cup__visible`
2. Full object track
   - Used for object replay during the action stage
   - Code: `load_object_tracks(...)`
   - It loads the full per-object arrays for `pose_world_wxyz` / `pose_world_matrix` / `visible`

### Where hand coordinates come from

The hand file `hand_detections_1.npz` contains both position-like fields and orientation-like fields, but in the current AnyGrasp pipeline the final execution target position does not come from the hand file.

Instead:

- Position target: from AnyGrasp candidate `translation`
- Orientation ranking reference: from hand file `*_gripper_rotation_matrix`

Detailed logic:

1. Read `left_gripper_valid/right_gripper_valid`
   - If the exact keyframe is invalid, use the nearest valid hand frame
2. Read `left_gripper_rotation_matrix/right_gripper_rotation_matrix`
3. Compute the rotation gap between that hand orientation and each AnyGrasp candidate `rotation_matrix`
4. Rank candidates by:
   - hand-orientation mismatch `rotation_distance_deg`
   - nearest-object distance `nearest_object_distance_m`
   - AnyGrasp raw score `score`

- nearest valid hand frame: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L619)
- candidate ranking: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L626)

### How AnyGrasp candidates are converted from camera frame to world frame

The `translation` and `rotation_matrix` fields in the AnyGrasp JSON are camera-frame grasps. Before execution they are converted into world-frame target poses.

The transform chain is:

1. Read the candidate camera-frame pose
   - `trans_cam = grasp["translation"]`
   - `rot_cam = grasp["rotation_matrix"]`
2. Apply optional orientation post-processing
   - `candidate_post_rot_xyz_deg`
   - `candidate_orientation_remap_label`
3. Use the current RoboTwin head-camera world pose as the extrinsic transform
4. Apply the `camera_cv_axis_mode` camera-axis conversion matrix
5. Obtain `raw_pose_world_wxyz`
6. If `--candidate_target_local_x_offset_m` is set
   - shift along the candidate's own local `+X`
   - producing the final planning/visualization pose `pose_world_wxyz`
7. If `--candidate_keep_camera_up 1` is enabled
   - only resolve the 180-degree roll ambiguity around the forward axis
   - so the camera top axis stays as upward as possible in world `+Z`

Core code:
- candidate to world conversion: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L443)
- camera to world conversion: [render_hand_retarget_r1_npz.py](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz.py#L1023)
- local X offset in world coordinates: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L435)

A simplified formula is:

```text
candidate_world = head_camera_world * camera_cv_axis_mode * candidate_cam
```

Where:
- `head_camera_world` comes from the current RoboTwin robot head-camera link pose, not from the replay file
- `camera_cv_axis_mode` converts the CV camera-axis convention into RoboTwin's internal local camera convention
- `candidate_cam` is the raw AnyGrasp output

### Where robot arm targets come from

The actual target sent to the arm is `CandidatePose.pose_world_wxyz`, meaning the world-frame TCP target after camera-to-world conversion, optional local-X offset, and optional roll disambiguation.

Execution uses three stages:

1. `pregrasp`
   - uses `pose_with_offset_along_local_x(grasp_pose, approach_offset_m)`
   - meaning: first move to an approach point that retreats backward along the target local X axis
2. `grasp`
   - move to keyframe-1 `candidate.pose_world_wxyz`
3. `action`
   - move to keyframe-2 `candidate.pose_world_wxyz`

### How objects move during the action stage

There are two branches controlled by `--replay_objects_during_action`.

#### Branch A: no replay, attach the object to TCP

When `--replay_objects_during_action 0`:

1. After grasping, read the current TCP world pose
2. Compute
   - `tcp_to_object = inv(tcp_world) @ object_world`
   - which freezes the object's pose relative to TCP
3. During the action stage, at every step compute
   - `object_world = tcp_world @ tcp_to_object`
   - and update the object actor

This behaves like rigid attachment to the hand.

Relevant code:
- single-arm attachment: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L2925)
- dual-arm attachment: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L2726)
- object update during execution: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L1947)

#### Branch B: replay world-space object tracks

When `--replay_objects_during_action 1`:

1. The object is not attached to TCP
2. The code creates an `ExecutionObjectReplayConfig`
   - `start_frame = keyframe1`
   - `end_frame = keyframe2`
   - `object_tracks =` the full tracks from `multi_object_world_poses.npz`
3. During action execution, given current progress `progress_01`
   - a source frame is obtained by linear interpolation
   - then the nearest replay frame in `selected_source_frame_indices` is selected
4. Every object actor is directly set to the world pose stored for that replay frame

This means the robot follows its own motion plan while the scene objects follow the replay file's world-space object trajectories.

### Practical follow-up usage

If you want to debug cases where the grasp looks right but the object and arm appear inconsistent, check in this order:

1. Inspect `plan_summary.json`
   - confirm the actual executed `selected_candidates[*].pose_world_wxyz`
   - confirm the candidate ids are the ones you manually specified
2. Inspect `selected_candidates[*].raw_pose_world_wxyz`
   - this tells you whether the issue comes from camera-to-world conversion or from `candidate_target_local_x_offset_m`
3. Cross-check `multi_object_world_poses.npz`
   - because in your command the action-stage object motion comes from replayed world tracks, not TCP attachment
4. If you want per-frame world-state traces during execution
   - enable `--save_pose_debug 1`
   - focus on `pose_debug.jsonl` fields such as `current_left_tcp_pose_world_wxyz`, `current_right_tcp_pose_world_wxyz`, and `object_actor_poses`
