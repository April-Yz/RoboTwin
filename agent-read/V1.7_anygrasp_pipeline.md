# V1.7 AnyGrasp Pipeline

## Scope

This document covers the RoboTwin-side workflow after AnyGrasp results have already been produced on another server.

Boundary:
- This repo does not run AnyGrasp inference as the normal pipeline entrypoint.
- It consumes AnyGrasp outputs and turns them into replay, debug, and execution results.

## Inputs

Each video id needs three inputs:

1. AnyGrasp results
   - `grasps/grasp_<frame>.json`
2. Replay result
   - `multi_object_world_poses.npz`
3. Hand result
   - `hand_detections_<id>.npz`

Typical roots:

- AnyGrasp:
  - `/home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_batch_results`
- Replay:
  - `/home/zaijia001/ssd/RoboTwin/code_painting/replay_m_obj_pose_d_pour_blue`
- Hand:
  - `/home/zaijia001/ssd/data/R1/gt_depth_vis/d_pour_blue/hand_vis`

## Main Scripts

- Batch runner: [run_plan_anygrasp_keyframes_r1_batch.sh](/home/zaijia001/ssd/RoboTwin/code_painting/run_plan_anygrasp_keyframes_r1_batch.sh)
- Batch planner: [plan_anygrasp_keyframes_r1_batch.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1_batch.py)
- Single-video planner: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py)

## Logic

1. Load grasp candidates for two keyframes.
2. Load hand orientations from `hand_detections_<id>.npz`.
3. Rank left/right candidates.
4. Optionally apply object-name and distance constraints.
5. Optionally accept manual candidate overrides.
6. Plan `pregrasp -> grasp -> action`.
7. Save videos, summaries, and rank preview images.

## Outputs

Per-video:

- `debug_selection_preview.mp4`
- `debug_execution_preview.mp4`
- `head_cam_plan.mp4`
- `rank_previews/keyframe_<frame>_rank_<k>.png`
- `plan_summary.json`

Batch:

- `batch_plan_summary.json`

## Key Parameters

### Keyframe and arm selection

- `--keyframes A B`: grasp keyframe and action keyframe
- `--arm auto|left|right`: execution arm
- `--execute_both_arms 1`: synchronized dual-arm execution

### Candidate filtering

- `--left_target_object`
- `--right_target_object`
- `--candidate_object_max_distance_m`
- `--enforce_target_object_constraint 0|1`
- `--enforce_candidate_distance_constraint 0|1`

### Candidate display and manual override

- `--debug_candidate_top_k`
- `--debug_common_candidate_top_k`
- `--debug_show_all_candidates 0|1`
- `--manual_candidate FRAME ARM CANDIDATE_IDX`

### Orientation debugging

- `--candidate_orientation_remap_label`
- `--candidate_post_rot_xyz_deg`

### Execution-quality debugging

- `--planner_backend urdfik|curobo`
- `--replan_until_reached 1`
- `--replan_until_reached_max_attempts`
- `--reach_error_pose_source tcp|ee`
- `--init_prefix_frames`

### Visualization

- `--save_debug_preview`
- `--save_debug_execution_preview`
- `--save_rank_preview_images`
- `--rank_preview_top_n`
- `--enable_viewer`
- `--viewer_wait_at_end`

## Visualization Semantics

- Green gripper: common candidate set
- Blue gripper: left-hand top candidates
- Orange gripper: right-hand top candidates
- Red gripper: selected candidate

Candidate ids are rendered as small plain numbers with no background box.

## How To Use The Outputs

- `debug_selection_preview.mp4`: decide whether candidate selection is reasonable
- `rank_previews/`: choose exact candidate ids manually
- `debug_execution_preview.mp4`: inspect whether the robot reaches keyframe A before keyframe B
- `plan_summary.json`: inspect selected candidates, stage status, and per-attempt histories
