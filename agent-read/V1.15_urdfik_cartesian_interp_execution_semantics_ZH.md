# V1.15 URDF IK Cartesian Interp Execution Semantics

## 结论

当前 `planner_backend=urdfik` 且 `urdfik_trajectory_mode=cartesian_interp_ik` 时：

- 中间 `ee/tcp pose` 插值确实会被计算
- 每个中间 waypoint 也确实会分别求一次 IK
- 但这些中间 waypoint **不会被逐段执行**
- 实际执行阶段仍然主要是：
  - `current_joints -> target_joints`
  - 做一次 joint-space 线性插值

所以这条链路的真实语义更接近：

- `cartesian_interp_ik = 用 Cartesian waypoint 帮助 IK 更稳定地找到最终 joint target`

而不是：

- `cartesian_interp_ik = 机器人严格沿着这些 Cartesian waypoint 逐段跟踪执行`

## 为什么会这样

### 1. 规划阶段确实生成了 Cartesian waypoint

在 `render_hand_retarget_r1_npz_urdfik.py` 中：

- `_plan_path_cartesian_interp_ik(...)`

它会：

1. 读取当前 TCP pose 和目标 TCP pose
2. 用 `_interpolate_tcp_pose_world_series(...)` 生成一串 TCP waypoint
3. 对每个 waypoint 调用 `_solve_ik_best_candidate(...)`
4. 得到一串 `joint_waypoints`

因此：

- `ee/tcp pose` 插值是存在的
- `joint_waypoints` 也是存在的

### 2. 但执行阶段没有消费完整 joint_waypoints

真正执行在同文件的：

- `execute_plans(...)`

这里并没有逐段执行 `plan["position"]` 或 `joint_waypoints`。

它当前做的是：

1. 读取 `current_joints`
2. 读取 `target_joints`
3. 对这两个端点做一次 `np.linspace(...)`
4. 直接把这条 joint-space 直线下发执行

所以：

- 中间 Cartesian waypoint 没有被当成实际执行路径
- 它们只影响“最终 `target_joints` 是怎么求出来的”

## 这会带来什么现象

### 1. `cartesian_interp_ik` 对解算结果有影响

因为中间 waypoint 会影响：

- 每步 IK 的 seed
- 最终解落入哪个局部 basin
- 最终 `target_joints` 是哪个解

所以它不是“完全没用”。

### 2. 但它对最终执行轨迹形状的影响有限

因为机器人没有真正按：

- `joint_waypoints[0] -> joint_waypoints[1] -> ... -> joint_waypoints[n]`

去执行。

而是只按：

- `current_joints -> final_target_joints`

去执行。

因此你看到的轨迹可能会出现：

- 不像真正的 Cartesian 路径
- 中间姿态变化和你预想的 waypoint 轨迹不一致
- 虽然开了 `cartesian_interp_ik`，但轨迹仍像 joint interpolation

## 自动 waypoint 模式和这个问题的关系

最近新增的：

- `--urdfik_cartesian_interp_steps -1`

只是改变了 waypoint 的数量选择方式：

- 小位移少插值
- 大位移多插值

它没有改变上面的根本事实：

- waypoint 仍然主要用于 IK 引导
- 当前执行仍不是严格逐段跟踪这些 waypoint

## 当前代码位置

规划与 waypoint 生成：

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `_plan_path_cartesian_interp_ik(...)`
  - `_interpolate_tcp_pose_world_series(...)`
  - `_solve_ik_best_candidate(...)`

实际执行：

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `execute_plans(...)`
  - `_execute_single_ik_plan(...)`

## 重新 debug 时应该先看什么

如果问题是：

- “IK 求不稳”
- “最终 joint 解方向不对”

那应该先看：

- Cartesian waypoint 生成
- 每个 waypoint 的 IK 解
- seed 如何传递

如果问题是：

- “轨迹看起来不像 Cartesian 路径”
- “执行出来不平滑”

那应该先看：

- `execute_plans(...)` 是否真正消费 `joint_waypoints`

这是当前执行语义的关键限制。
