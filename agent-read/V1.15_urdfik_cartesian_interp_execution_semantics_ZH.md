# V1.15 URDF IK Cartesian Interp Execution Semantics

## 结论

在最新修正后，当前 `planner_backend=urdfik` 且 `urdfik_trajectory_mode=cartesian_interp_ik` 时：

- 中间 `ee/tcp pose` 插值会被计算
- 每个中间 waypoint 都会分别求一次 IK
- 得到的 `joint_waypoints` 会被真正逐段执行

所以这条链路现在的真实语义是：

- `cartesian_interp_ik = 先用 Cartesian waypoint 逐点求 IK，再按生成的 joint waypoint 序列执行`

## 为什么会这样

### 1. 规划阶段确实生成了 Cartesian waypoint

在 `render_hand_retarget_r1_npz_urdfik.py` 中：

- `_plan_path_cartesian_interp_ik(...)`

它会：

1. 读取当前 TCP pose 和目标 TCP pose
2. 用 `_interpolate_tcp_pose_world_series(...)` 生成一串 TCP waypoint
3. 对每个 waypoint 调用 `_solve_ik_best_candidate(...)`
4. 得到一串 `joint_waypoints`

因此：

- `ee/tcp pose` 插值是存在的
- `joint_waypoints` 也是存在的

### 2. 现在执行阶段会消费完整 joint_waypoints

真正执行在同文件的：

- `execute_plans(...)`
- `_execute_single_ik_plan(...)`

现在会直接消费：

- `plan["position"]`
- `plan["velocity"]`

对双臂情况：

- 左右臂按相对轨迹进度交错推进

对单臂情况：

- 直接逐段执行该 arm 的整条 `position` 轨迹

所以：

- 中间 Cartesian waypoint 求出的 `joint_waypoints` 现在会成为实际执行路径的一部分

## 这会带来什么现象

### 1. `cartesian_interp_ik` 对解算结果有影响

因为中间 waypoint 会影响：

- 每步 IK 的 seed
- 最终解落入哪个局部 basin
- 最终 `target_joints` 是哪个解

所以它不是“完全没用”。

### 2. 现在它会更直接影响执行轨迹形状

因为机器人现在会真正按：

- `joint_waypoints[0] -> joint_waypoints[1] -> ... -> joint_waypoints[n]`

逐段推进。

因此：

- `cartesian_interp_ik` 不再只是“帮助求最终解”
- 它也会更直接地影响最终执行轨迹

## 自动 waypoint 模式和这个问题的关系

最近新增的：

- `--urdfik_cartesian_interp_steps -1`

只是改变了 waypoint 的数量选择方式：

- 小位移少插值
- 大位移多插值

它不会改变新的根本事实：

- waypoint 现在既用于 IK 引导，也用于实际执行

## 当前代码位置

规划与 waypoint 生成：

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `_plan_path_cartesian_interp_ik(...)`
  - `_interpolate_tcp_pose_world_series(...)`
  - `_solve_ik_best_candidate(...)`

实际执行：

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `execute_plans(...)`
  - `_execute_single_ik_plan(...)`

## 为什么 viewer 里看起来是平滑插值，但 `head_cam_plan.mp4` 里像瞬移

这条执行链里其实同时存在两种“时钟”：

1. **physics / viewer 时钟**
2. **视频录制时钟**

它们相关，但并不完全相同。

### 1. `urdfik_cartesian_interp_steps` 属于规划层

在 `render_hand_retarget_r1_npz_urdfik.py` 的：

- `_plan_path_cartesian_interp_ik(...)`

`--urdfik_cartesian_interp_steps` 控制的是：

- 从当前 TCP pose 到目标 TCP pose，中间生成多少个 **Cartesian TCP waypoint**

所以当你设置：

- `--urdfik_trajectory_mode cartesian_interp_ik`
- `--urdfik_cartesian_interp_steps 20`

含义是：

- 先生成大约 20 个 TCP waypoint
- 对每个 waypoint 逐点求 IK
- 最后得到一条多点的 joint trajectory

它是一个**规划密度**参数，不是执行末尾等待参数。

### 2. `execute_interp_steps` 还可能再把 joint trajectory 加密一次

在 `plan_anygrasp_keyframes_r1.py` 中：

- `interpolate_joint_trajectory(...)`
- `execute_single_arm_plan(...)`
- `execute_dual_arm_plan(...)`

执行前会先调用：

- `interpolate_joint_trajectory(position, velocity, execute_interp_steps)`

也就是说：

- 即使规划阶段已经产生了 20 个 joint waypoint
- 执行器仍然可能再把这条 joint 轨迹补得更密

需要注意：

- 如果规划轨迹本来点就很多，`execute_interp_steps` 只会轻度加密
- 如果规划轨迹点很少，真正决定你看到多少执行步的往往是 `execute_interp_steps`

例如：

- 规划大约有 20 个点
- `--execute_interp_steps 24`

那最终执行的 command 点数仍可能比 20 更多。

### 3. `joint_command_scene_steps` 控制每个 waypoint 实际给多少个 physics step

在 `execute_single_arm_plan(...)` / `execute_dual_arm_plan(...)` 中：

每个 joint waypoint 的执行顺序是：

1. 下发一个 joint target
2. 调用 `renderer.step_scene(steps=joint_command_scene_steps)`
3. 调用 `record_frame(...)` 录一帧视频

所以导出视频的主节奏其实是：

- **每个 command waypoint 录 1 帧**

而不是：

- 每个 physics 子步都录 1 帧

### 4. `joint_target_wait_steps` 是轨迹下发完成后的“末端收敛等待”

当所有 command waypoint 都跑完后，代码会调用：

- `settle_arms_to_targets(...)`

这个函数会：

- 检查当前 joint 与最终目标 joint 的误差
- 如果误差仍大于 `joint_target_wait_tol_rad`
- 就继续额外 step physics，最多 step `joint_target_wait_steps` 次

这一段非常关键：

- viewer 在这些额外 `step_scene(steps=1)` 期间仍然会持续 render
- 但是 `head_cam_plan.mp4` **不会把这段 settle loop 的每一步都录下来**
- 主视频只会在 settle loop 结束后再录一帧 `plan_step=done`

所以如果机械臂的真实跟随主要发生在这段 settle loop 里，就会出现：

- **viewer：** 你看到它继续平缓移动
- **导出 head 视频：** 你只看到 settle 前一帧和 settle 后最后一帧
- 肉眼效果就像“中间直接瞬移过去了”

这就是你现在看到 viewer 平滑、但 head cam 视频像瞬移的最主要原因。

### 5. 你这条命令当前意味着什么

你给的命令里关键参数是：

- `--urdfik_cartesian_interp_steps 20`
- `--joint_target_wait_steps 70`
- `--viewer_frame_delay 0.02`

它们组合起来的实际含义是：

- 规划层会生成比较密的 Cartesian waypoint
- 执行层会把这些 waypoint 真正下发出去（而且可能还会再轻微插值）
- 在全部 waypoint 下发完之后，还允许最多 70 个额外 physics step 去等机械臂收敛
- viewer 会把这 70 个额外 step 也显示出来
- 但保存的 `head_cam_plan.mp4` 不会逐帧记录这 70 个额外 step

所以一个非常常见的现象就是：

- viewer 看起来是连续过去的
- 导出视频里像是在阶段末尾突然靠近目标

### 6. 会影响这个现象的参数

最相关的参数有：

- `--urdfik_cartesian_interp_steps`
  - 规划层 Cartesian TCP waypoint 数
- `--execute_interp_steps`
  - 执行前 joint 轨迹再次插值的密度
- `--joint_command_scene_steps`
  - 每个 command waypoint 给多少个 physics step
- `--joint_target_wait_steps`
  - command 轨迹结束后，最多再等多少个 physics step 让关节收敛
- `--joint_target_wait_tol_rad`
  - joint 误差收敛到多小就可以提前停止等待
- `--settle_steps`
  - 进入最终 joint 收敛循环前，先额外停多少个仿真步
- `--viewer_frame_delay`
  - 只影响人眼在 viewer 里看起来多慢，不会增加导出视频帧数
- `--fps`
  - 只影响导出视频的播放速度，不决定 settle loop 录了多少帧

### 7. 实际上该怎么理解这两个参数

如果你想理解“为什么 viewer 里有平滑移动，但导出视频里像跳过去”，关键不能只看规划插值密度，还要看：

- 机械臂真正完成跟随动作，是主要发生在“command loop”里
- 还是主要发生在后面的“未逐帧录制 settle loop”里

一句话概括就是：

- `urdfik_cartesian_interp_steps` 决定**规划路径有多密**
- `joint_target_wait_steps` 决定**轨迹下发完后，还给机械臂多少额外时间慢慢追到最终关节目标**

如果 controller 响应相对慢，而 `joint_command_scene_steps` 又比较小，那么大量肉眼可见的运动就会从“已录制的 command loop”转移到“viewer 可见但主视频没逐帧录的 settle loop”。

## 重新 debug 时应该先看什么

如果问题是：

- “IK 求不稳”
- “最终 joint 解方向不对”

那应该先看：

- Cartesian waypoint 生成
- 每个 waypoint 的 IK 解
- seed 如何传递

如果问题是：

- “轨迹看起来不像 Cartesian 路径”
- “执行出来不平滑”

那现在应该先看：

- waypoint 数是否过多或过少
- `joint_waypoints` 本身是否抖动
- 双臂交错执行是否带来节奏问题

当前已经不再是“执行层完全忽略 joint_waypoints”的旧问题。
