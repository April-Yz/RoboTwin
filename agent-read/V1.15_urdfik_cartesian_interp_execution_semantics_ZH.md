# V1.15 URDF IK Cartesian Interp Execution Semantics

## 结论

在最新修正后，当前 `planner_backend=urdfik` 且 `urdfik_trajectory_mode=cartesian_interp_ik` 时：

- 中间 `ee/tcp pose` 插值会被计算
- 每个中间 waypoint 都会分别求一次 IK
- 得到的 `joint_waypoints` 会被真正逐段执行

所以这条链路现在的真实语义是：

- `cartesian_interp_ik = 先用 Cartesian waypoint 逐点求 IK，再按生成的 joint waypoint 序列执行`

## 为什么会这样

### 1. 规划阶段确实生成了 Cartesian waypoint

在 `render_hand_retarget_r1_npz_urdfik.py` 中：

- `_plan_path_cartesian_interp_ik(...)`

它会：

1. 读取当前 TCP pose 和目标 TCP pose
2. 用 `_interpolate_tcp_pose_world_series(...)` 生成一串 TCP waypoint
3. 对每个 waypoint 调用 `_solve_ik_best_candidate(...)`
4. 得到一串 `joint_waypoints`

因此：

- `ee/tcp pose` 插值是存在的
- `joint_waypoints` 也是存在的

### 2. 现在执行阶段会消费完整 joint_waypoints

真正执行在同文件的：

- `execute_plans(...)`
- `_execute_single_ik_plan(...)`

现在会直接消费：

- `plan["position"]`
- `plan["velocity"]`

对双臂情况：

- 左右臂按相对轨迹进度交错推进

对单臂情况：

- 直接逐段执行该 arm 的整条 `position` 轨迹

所以：

- 中间 Cartesian waypoint 求出的 `joint_waypoints` 现在会成为实际执行路径的一部分

## 这会带来什么现象

### 1. `cartesian_interp_ik` 对解算结果有影响

因为中间 waypoint 会影响：

- 每步 IK 的 seed
- 最终解落入哪个局部 basin
- 最终 `target_joints` 是哪个解

所以它不是“完全没用”。

### 2. 现在它会更直接影响执行轨迹形状

因为机器人现在会真正按：

- `joint_waypoints[0] -> joint_waypoints[1] -> ... -> joint_waypoints[n]`

逐段推进。

因此：

- `cartesian_interp_ik` 不再只是“帮助求最终解”
- 它也会更直接地影响最终执行轨迹

## 自动 waypoint 模式和这个问题的关系

最近新增的：

- `--urdfik_cartesian_interp_steps -1`

只是改变了 waypoint 的数量选择方式：

- 小位移少插值
- 大位移多插值

它不会改变新的根本事实：

- waypoint 现在既用于 IK 引导，也用于实际执行

## 当前代码位置

规划与 waypoint 生成：

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `_plan_path_cartesian_interp_ik(...)`
  - `_interpolate_tcp_pose_world_series(...)`
  - `_solve_ik_best_candidate(...)`

实际执行：

- `code_painting/render_hand_retarget_r1_npz_urdfik.py`
  - `execute_plans(...)`
  - `_execute_single_ik_plan(...)`

## 重新 debug 时应该先看什么

如果问题是：

- “IK 求不稳”
- “最终 joint 解方向不对”

那应该先看：

- Cartesian waypoint 生成
- 每个 waypoint 的 IK 解
- seed 如何传递

如果问题是：

- “轨迹看起来不像 Cartesian 路径”
- “执行出来不平滑”

那现在应该先看：

- waypoint 数是否过多或过少
- `joint_waypoints` 本身是否抖动
- 双臂交错执行是否带来节奏问题

当前已经不再是“执行层完全忽略 joint_waypoints”的旧问题。
