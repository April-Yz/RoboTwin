# V1.11 AnyGrasp 目标位姿局部 X 偏移

## 问题定义

当前 AnyGrasp 关键帧链路里，planner 侧使用的 target pose 语义可能有歧义：

- 原始 AnyGrasp candidate 可能更像“指尖 TCP pose”
- 也可能更像“手腕 / 最后一个 link 的 pose”

这件事很关键，因为 `urdfik` 不是直接拿输入的 world pose 去解。

当前调用链是：

1. `plan_anygrasp_keyframes_r1.py` 先构造 world-space candidate pose
2. `renderer.plan_path(arm, target_pose_world)`
3. `render_hand_retarget_r1_npz_urdfik.py::_target_tcp_world_to_ee_base(...)`
4. `robot._trans_from_gripper_to_endlink(...)`
5. 最后对 `left_gripper_link` / `right_gripper_link` 做 IK

所以真正求解的目标是 endlink/EE pose，不是原始输入 pose 本身。

## 现有代码已经能证明什么

从当前实现可以直接确认：

- planner 把传入的 AnyGrasp pose 当成 gripper/TCP 风格的目标
- `_trans_from_gripper_to_endlink(...)` 再把它转成 wrist/endlink 约定
- 这里面大约会沿夹爪局部 `+X` 再加 `0.12 m`

相关代码：

- [`envs/robot/robot.py`](/home/zaijia001/ssd/RoboTwin/envs/robot/robot.py)
- [`code_painting/render_hand_retarget_r1_npz_urdfik.py`](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz_urdfik.py)
- [`code_painting/plan_anygrasp_keyframes_r1.py`](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py)

这也说明你这次怀疑是成立的：

- 如果当前这套 AnyGrasp raw candidate 在你的场景里其实更像 wrist/endlink pose，
- 那么现有这层 TCP->EE 转换就会把 IK 目标继续往前推，
- 这时 planner 侧必须先把 target 往后退回来。

## 新参数

新增：

- `--candidate_target_local_x_offset_m`

语义：

- 在规划前、也在目标坐标轴可视化前
- 先沿夹爪局部 `+X` 对 candidate target pose 做一个额外平移

示例：

- `0.0`
  - 保持现在的行为
- `-0.12`
  - 沿局部 `+X` 反方向后退 `12 cm`
  - 如果你确认当前 raw candidate 更像 wrist/endlink pose，而你想让 planner 输入改成指尖 TCP target，就用这个

## 为什么不直接把一种解释写死

raw AnyGrasp pose 到底更像：

- TCP
- 还是 wrist/endlink

会受这些因素影响：

- 上游使用的 gripper / mesh 约定
- grasp frame 的局部定义
- RoboTwin 当前 TCP vs endlink 的约定

如果把一种解释硬编码死，后面会很脆。

所以这次改成显式参数更稳：

- 需要原始 candidate 时，用 `0.0`
- 需要“回退成指尖 target”时，用 `-0.12`

不用再改 planner 主逻辑。

## 这次 planner 里具体改了什么

现在每个 candidate 会同时保存两套 pose：

- `raw_pose_world_wxyz`
  - 经过 orientation remap / camera-up 处理后，直接从 AnyGrasp 转过来的世界坐标 pose
- `pose_world_wxyz`
  - 再叠加 `--candidate_target_local_x_offset_m` 之后，真正用于规划和目标轴可视化的 pose

所以现在 viewer 里的 target axis 显示的是：

- 实际送进 planner 的 target pose

不再只是原始导入 candidate。

`plan_summary.json` 现在也会额外记录：

- 顶层 `candidate_target_local_x_offset_m`
- 每个 candidate 的 `raw_pose_world_wxyz`
- 每个 candidate 的 `pose_world_wxyz`

## 现在 plan 的函数名是什么，实际求解目标是什么

当前关键帧规划里真正用到的函数名是：

- keyframe workflow 入口：
  - `HandRetargetR1URDFIKRenderer.plan_path(...)`
- 内部目标转换：
  - `_target_tcp_world_to_ee_base(...)`
- 最后真正求解 IK：
  - `URDFInverseKinematics.solve_ik(...)`

现在实际求解的目标是：

- `left_gripper_link` 或 `right_gripper_link` 的目标 pose
- 坐标系是 robot base frame

不是直接对 raw AnyGrasp world pose 本身求解。

## 推荐指令

如果你现在确认当前 raw candidate 更像 wrist/endlink pose，而你希望 planner 输入改成指尖 TCP target，就加：

```bash
--candidate_target_local_x_offset_m -0.12
```
