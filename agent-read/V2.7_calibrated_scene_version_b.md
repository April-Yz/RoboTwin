# V2.7 Calibrated Scene Version B

## Goal

Use version B alignment:

- remove the earlier `+90°` anchor rotation on the first robot
- keep the calibration bundle's left/right spread aligned with world `y` as much as possible
- rotate the table convention instead of rotating the anchor robot
- store outputs under `code_painting/pika/`

## Why version B

Earlier versions used a manually added quaternion on the first anchor robot:

- `[0.70710678, 0.0, 0.0, 0.70710678]`

That rotated the anchor by `+90°` around `z`, which made the calibration bundle's intrinsic left/right `y` spread appear as world `x` spread.

Version B removes that anchor rotation and instead rotates the table convention by 90 degrees.

## Alignment with R1 / UR intuition

This version is closer to the existing R1-style intuition:

- left/right spread is treated primarily as a `y`-axis relation
- the first robot keeps identity world orientation
- the table long edge is treated as running along world `y`
- robots face toward world `+x`

## Reused vs not reused from `robot_config_PiperPika_agx_dual_table.json`

### Reused

- the magnitude of the previous tabletop edge inset:
  - current config uses `y = -0.25`
  - version B reuses `|y| = 0.25` as the new world-`x` inset
- tabletop height:
  - `z = 0.75`

### Not reused / intentionally ignored

- the previous quaternion in `robot_pose`
  - ignored because version B removes the earlier `+90°` anchor rotation
- `embodiment_dis`
  - not used for calibrated scene placement
  - calibration directly provides the second robot pose from `left_base_T_right_base`
- `right_embodiment_config`
  - not used in the calibrated scene script
  - the second robot pose comes from the calibration bundle, not from the dual-instance split config

## Version B anchor construction

Using the current config and the bundle:

- left anchor world `x` = `-abs(config.left_embodiment_config.robot_pose[0][1])`
- left anchor world `y` = `-0.5 * left_base_T_right_base.translation.y`
- left anchor world `z` = config tabletop height
- left anchor quaternion = identity

This produced:

### Left base
- position: `[-0.25, 0.285920, 0.75]`
- quaternion (wxyz): `[1.0, 0.0, 0.0, 0.0]`

### Right base
- position: `[-0.273641, -0.285920, 0.770517]`
- quaternion (wxyz): `[0.999961, 0.006516, 0.003163, -0.005038]`

### Head camera
- position: `[-0.142118, 0.016533, 1.214396]`
- quaternion (wxyz): `[0.854012, 0.012553, 0.518857, -0.035978]`

So this is not the same as the earlier overview used in version A.
