# V1.12 Joint Gripper + Object Replay Logic

## 1. Goal

This path describes the current combined execution mode:

1. AnyGrasp selects left/right targets for keyframe 1 and keyframe 22.
2. The robot executes `pregrasp -> grasp -> close gripper -> action`.
3. During `action`, `bottle/cup` are no longer attached to the TCP. They are replayed from `replay_dir/multi_object_world_poses.npz` in world coordinates.

This is the current formal logic for "gripper execution + object trajectory replay together".

## 2. Upstream inputs

### 2.1 Hand / AnyGrasp inputs

- `hand_npz`: `hand_detections_<id>.npz`
- `anygrasp_dir`: `grasp_*.json` / `vis/*`

Roles:

- `hand_npz` provides the hand reference orientation used for candidate ranking.
- `anygrasp_dir` provides per-frame grasp candidates, which are then filtered, ranked, optionally overridden manually, and finally selected as keyframe targets.

### 2.2 Object trajectory input

- `replay_dir/multi_object_world_poses.npz`

Source:

- Usually exported by `render_multi_object_pose_r1_npz.py`.
- This file already stores world-frame object poses, not raw camera-frame poses.

Important fields:

- `frame_indices`
- `<object>__pose_world_wxyz`
- `<object>__pose_world_matrix`
- `<object>__visible`
- `head_camera_pose_world_wxyz`

## 3. How objects enter the simulator

### 3.1 Initial keyframe object state

Before planner execution starts:

1. load `ObjectState` for the frame corresponding to `keyframe_1`
2. create one execution actor per object
3. place each actor at its keyframe-1 world pose

Code locations:

- `load_object_states(...)`
- `create_execution_object_actor(...)`
- main flow using `object_states_per_frame[keyframes[0]]`

### 3.2 Action-stage object replay

When enabled:

- `--replay_objects_during_action 1`

behavior becomes:

1. During `pregrasp` / `grasp` / `pause_after_keyframe1`:
   - objects stay at keyframe-1 world pose
2. During `action`:
   - build `ExecutionObjectReplayConfig`
   - store `start_frame=keyframe_1`, `end_frame=keyframe_2`
   - map current action progress to a replay source frame
   - set each object actor to the replay pose from that frame

Code locations:

- `load_object_tracks(...)`
- `ExecutionObjectReplayConfig`
- `apply_object_replay_for_progress(...)`
- `execute_dual_arm_plan(...)` / `execute_single_arm_plan(...)`

### 3.3 Why the object no longer follows the gripper

The old attach-to-TCP behavior is only used when:

- `--replay_objects_during_action 0`

Once this is enabled:

- `--replay_objects_during_action 1`

object actors are explicitly overwritten by replay poses during `action`, so they no longer move with the gripper.

## 4. How gripper keyframe targets are defined

Each arm gets two targets:

- `keyframe_1`
- `keyframe_2`

Execution constructs:

- `pregrasp_targets`: keyframe-1 target shifted by `approach_offset_m` along local `+X`
- `grasp_targets`: keyframe-1 target
- `action_targets`: keyframe-2 target

Stage mapping:

1. `pregrasp`: approach keyframe-1 target
2. `grasp`: reach keyframe-1 target and close gripper
3. `pause_after_keyframe1`: optional hold
4. `action`: move to keyframe-2 target while replaying object tracks

## 5. Current camera-up / roll rule

### 5.1 Previous version

Previously, `--candidate_keep_camera_up 1` did this for every keyframe independently:

- compare two equivalent roll states of the same grasp:
  - original pose
  - pose rolled by `180` degrees around local `+X`
- select whichever keeps the top axis more upward

This solves the upside-down ambiguity, but later keyframes can still spin a full turn around the forward axis.

### 5.2 Current version

Now the rule is sequential across selected keyframes:

1. First keyframe:
   - still uses the "more upward" rule to define the roll baseline
2. Later keyframes:
   - still compare only the two equivalent roll states
   - but selection is changed to:
     - choose the variant with the smallest rotation change relative to the previously selected keyframe
   - if the difference is nearly tied, prefer the one with the higher upward-facing score

Result:

- keyframe 1 keeps the gripper camera / top side upward overall
- keyframe 22 no longer makes a large extra roll just to satisfy the upward rule; it instead preserves roll continuity relative to keyframe 1

Related debug field:

- `camera_up_selection_mode`
  - `keyframe1_keep_up`
  - `follow_previous_base`
  - `follow_previous_flip180`
