# V1.12 物体与夹爪联合 Replay 逻辑

## 1. 目标

这条链路描述的是：

1. 用 AnyGrasp 选出第 1 帧和第 22 帧的左右手抓取目标。
2. 机器人按照 `pregrasp -> grasp -> close gripper -> action` 执行关键帧。
3. 在 `action` 阶段，`bottle/cup` 不再附着到 TCP，而是按 `replay_dir/multi_object_world_poses.npz` 里的世界系轨迹回放。

这就是当前“夹爪执行 + 物体轨迹一起播放”的正式逻辑。

相关主入口：

- `/home/zaijia001/ssd/RoboTwin/code_painting/run_plan_anygrasp_keyframes_r1_batch.sh`
- `/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1_batch.py`
- `/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py`

## 2. 上游输入

### 2.1 手部 / AnyGrasp 输入

- `hand_npz`: `hand_detections_<id>.npz`
- `anygrasp_dir`: `grasp_*.json` / `vis/*`

作用：

- `hand_npz` 提供手部参考朝向，用于左右手候选排序。
- `anygrasp_dir` 提供每一帧的抓取候选，后续会被过滤、排序、手工覆盖、再选择成关键帧目标。

### 2.2 物体轨迹输入

- `replay_dir/multi_object_world_poses.npz`

来源：

- 一般由 `render_multi_object_pose_r1_npz.py` 生成。
- 这个文件里保存的已经是世界系物体 pose，不是原始相机系 pose。

主要字段：

- `frame_indices`
- `<object>__pose_world_wxyz`
- `<object>__pose_world_matrix`
- `<object>__visible`
- `head_camera_pose_world_wxyz`

## 3. 物体怎么接入模拟器

### 3.1 初始关键帧物体状态

在 planner 开始执行前：

1. 从 `keyframe_1` 对应帧加载 `ObjectState`
2. 为每个物体创建执行期 actor
3. 把 actor 先放到 `keyframe_1` 的世界系 pose

代码位置：

- `load_object_states(...)`
- `create_execution_object_actor(...)`
- 主流程中 `object_states_per_frame[keyframes[0]]`

### 3.2 action 阶段物体回放

当打开：

- `--replay_objects_during_action 1`

逻辑变成：

1. `pregrasp` / `grasp` / `pause_after_keyframe1` 阶段：
   - 物体保持在关键帧 1 的世界系 pose
2. `action` 阶段：
   - 构造 `ExecutionObjectReplayConfig`
   - 记录 `start_frame=keyframe_1`, `end_frame=keyframe_2`
   - 每次执行帧根据当前 `action` 进度映射到 replay 的 source frame
   - 再把每个物体 actor 设置到该 replay pose

代码位置：

- `load_object_tracks(...)`
- `ExecutionObjectReplayConfig`
- `apply_object_replay_for_progress(...)`
- `execute_dual_arm_plan(...)` / `execute_single_arm_plan(...)`

### 3.3 为什么物体不会再跟着夹爪走

只有当：

- `--replay_objects_during_action 0`

时，代码才会走“附着物体到 TCP”的旧逻辑。

一旦打开：

- `--replay_objects_during_action 1`

则 `action` 阶段明确使用 replay track 覆盖物体 actor pose，因此物体不再被夹爪带着移动。

## 4. 夹爪关键帧目标怎么定

每只手会分别得到两帧目标：

- `keyframe_1`
- `keyframe_2`

执行时会构造：

- `pregrasp_targets`: 在关键帧 1 的 target 上沿局部 `+X` 偏移 `approach_offset_m`
- `grasp_targets`: 关键帧 1 target
- `action_targets`: 关键帧 2 target

对应关系：

1. `pregrasp`: 先靠近第 1 帧目标
2. `grasp`: 到第 1 帧目标并闭合夹爪
3. `pause_after_keyframe1`: 可选停留
4. `action`: 去第 2 帧目标，同时可让物体按轨迹回放

## 5. 当前“朝上”规则

### 5.1 之前的版本

之前 `--candidate_keep_camera_up 1` 的逻辑是：

- 比较同一个 grasp 候选的两个等价 roll：
  - 原姿态
  - 绕局部 `+X` 翻转 `180` 度后的姿态
- 选“顶部轴更朝上”的那个

这会解决“上下颠倒”问题，但会导致后续关键帧有时绕前进轴转一大圈。

### 5.2 当前版本

现在规则改成了按关键帧顺序处理：

1. 第 1 个关键帧：
   - 仍然用“更朝上”的规则确定基准 roll
2. 后续关键帧：
   - 仍然只在两种等价 roll（原姿态 / 绕局部 `+X` 翻 180 度）之间选
   - 但选择标准改成：
     - 和前一个已选关键帧的旋转差最小
   - 若两者差不多，再偏向顶部轴更朝上的那个

结果：

- 第 1 帧保证相机 / 顶部整体朝上
- 第 22 帧不再为了“朝上”额外转大圈，而是优先保持和第 1 帧的 roll 连续性

相关字段：

- `camera_up_selection_mode`
  - `keyframe1_keep_up`
  - `follow_previous_base`
  - `follow_previous_flip180`
