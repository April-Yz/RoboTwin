# V1.7 AnyGrasp 流程

## 范围

这份文档说明 AnyGrasp 结果已经在别的服务器上跑完之后，RoboTwin 这一侧怎么继续做候选选择、可视化和关键帧执行。

边界：
- 这套流程默认不在 RoboTwin 里跑 AnyGrasp 推理
- RoboTwin 负责消费 AnyGrasp 结果并做执行

## 输入

每个视频 id 需要三类输入：

1. AnyGrasp 结果
   - `grasps/grasp_<frame>.json`
2. 物体 replay 结果
   - `multi_object_world_poses.npz`
3. 手部结果
   - `hand_detections_<id>.npz`

典型根目录：

- AnyGrasp：
  - `/home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_batch_results`
- Replay：
  - `/home/zaijia001/ssd/RoboTwin/code_painting/replay_m_obj_pose_d_pour_blue`
- Hand：
  - `/home/zaijia001/ssd/data/R1/gt_depth_vis/d_pour_blue/hand_vis`

## 主要脚本

- 批处理入口: [run_plan_anygrasp_keyframes_r1_batch.sh](/home/zaijia001/ssd/RoboTwin/code_painting/run_plan_anygrasp_keyframes_r1_batch.sh)
- 批处理调度: [plan_anygrasp_keyframes_r1_batch.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1_batch.py)
- 单视频实现: [plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py)

## 逻辑

1. 读取两个关键帧的抓取候选
2. 读取 `hand_detections_<id>.npz` 里的手部朝向
3. 给左右手候选分别排序
4. 可选加上物体约束和距离约束
5. 可选接受手工指定候选
6. 在 RoboTwin 里执行 `pregrasp -> grasp -> action`
7. 保存视频、总结和 rank preview 图片

### 手部坐标的数据来源

手部文件 `hand_detections_1.npz` 里有位置和朝向，但当前 AnyGrasp 管线对“目标执行位姿”的位置来源并不是 hand 文件，而是：

- 位置：来自 AnyGrasp 候选 `translation`
- 朝向排序参考：来自 hand 文件 `*_gripper_rotation_matrix`

具体逻辑：

1. 读取 `left_gripper_valid/right_gripper_valid`
   - 如果关键帧本身无效，就找最近的有效手部帧
2. 读取该帧的 `left_gripper_rotation_matrix/right_gripper_rotation_matrix`
3. 计算它与每个 AnyGrasp 候选 `rotation_matrix` 的旋转差
4. 按以下优先级排序候选：
   - 先看与手部朝向的角度差 `rotation_distance_deg`
   - 再看候选与最近物体的距离 `nearest_object_distance_m`
   - 最后看 AnyGrasp 原始分数 `score`

### AnyGrasp 候选如何从相机系变成世界系

AnyGrasp JSON 中的 `translation` 和 `rotation_matrix` 是相机坐标系下的抓取候选。真正执行前会先变成世界坐标系。

转换链条如下：

1. 取出候选相机系位姿
   - `trans_cam = grasp["translation"]`
   - `rot_cam = grasp["rotation_matrix"]`
2. 先做朝向后处理
   - `candidate_post_rot_xyz_deg`
   - `candidate_orientation_remap_label`
3. 用当前 RoboTwin 中 head camera 的世界位姿做外参变换
4. 再乘上 `camera_cv_axis_mode` 对应的相机轴变换矩阵
5. 得到候选世界位姿 `raw_pose_world_wxyz`
6. 如果设了 `--candidate_target_local_x_offset_m`
   - 会沿候选自身局部 `+X` 再平移
   - 得到真正用于规划和可视化的 `pose_world_wxyz`
7. 如果设了 `--candidate_keep_camera_up 1`
   - 只在保留抓取前向方向的前提下，解决绕前向轴的 180 度翻转二义性
   - 让相机 top 轴尽量朝世界 `+Z`

核心代码：
- 候选转世界系：[plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L443)
- 相机系到世界系：[render_hand_retarget_r1_npz.py](/home/zaijia001/ssd/RoboTwin/code_painting/render_hand_retarget_r1_npz.py#L1023)
- 世界坐标局部 X 偏移：[plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L435)

可以把它写成一个简化公式：

```text
candidate_world = head_camera_world * camera_cv_axis_mode * candidate_cam
```

其中：
- `head_camera_world` 来自 RoboTwin 当前机器人头部相机 link 的世界位姿，不来自 replay 文件
- `camera_cv_axis_mode` 负责把 CV 相机轴定义转成 RoboTwin 内部使用的相机局部轴定义
- `candidate_cam` 是 AnyGrasp 输出

### 机器人手臂目标坐标是怎么来的

手臂真正吃到的目标是 `CandidatePose.pose_world_wxyz`，也就是处理完坐标系变换、局部 X 偏移、可选 roll 修正之后的世界系 TCP 目标。

执行时分三个阶段：

1. `pregrasp`
   - 用 `pose_with_offset_along_local_x(grasp_pose, approach_offset_m)`
   - 含义：沿目标局部 X 反向退一段距离，先到预抓取点
2. `grasp`
   - 直接到关键帧 1 的 `candidate.pose_world_wxyz`
3. `action`
   - 到关键帧 2 的 `candidate.pose_world_wxyz`

### Action 阶段物体是怎么跟着走的

这里有两条分支，由 `--replay_objects_during_action` 控制。

#### 分支 A：不 replay，直接把物体挂到 TCP 上

当 `--replay_objects_during_action 0`：

1. 在抓取完成后，读取当前 TCP 世界位姿
2. 用
   - `tcp_to_object = inv(tcp_world) @ object_world`
   - 固定住“物体相对于 TCP 的相对位姿”
3. 后续 action 过程中，每一步都用
   - `object_world = tcp_world @ tcp_to_object`
   - 更新物体 actor

这相当于“刚体焊接到手上”。

对应代码：
- 单臂附着：[plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L2925)
- 双臂附着：[plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L2726)
- 执行时按 TCP 更新物体：[plan_anygrasp_keyframes_r1.py](/home/zaijia001/ssd/RoboTwin/code_painting/plan_anygrasp_keyframes_r1.py#L1947)

#### 分支 B：使用 replay 世界轨迹

当 `--replay_objects_during_action 1`：

1. 不再把物体焊到 TCP
2. 构造 `ExecutionObjectReplayConfig`
   - `start_frame = keyframe1`
   - `end_frame = keyframe2`
   - `object_tracks = multi_object_world_poses.npz` 中整段轨迹
3. 执行 action 时，根据当前规划进度 `progress_01`
   - 在线性插值得到一个 source frame
   - 再从 `selected_source_frame_indices` 中找最近 replay 帧
4. 直接把每个物体 actor 设置到该 replay 帧对应的世界位姿

这相当于“机器人按自己的规划运动，但场景物体按 replay 文件中的世界轨迹重放”。

### 你的这条命令里，各坐标数据实际来自哪里

按你给的命令：

- 手臂目标位姿来源
  - 不是 replay 文件
  - 不是 hand 文件的位置
  - 而是 `anygrasp_batch_results/d_pour_blue_1/grasps/grasp_000001.json` 和 `grasp_000022.json` 中你手工指定的 candidate
  - 然后再经过 `camera -> world` 变换、局部 X 偏移、可选 roll 处理
- 手部参考朝向来源
  - `hand_detections_1.npz` 的 `left_gripper_rotation_matrix/right_gripper_rotation_matrix`
  - 只用于排序和挑 candidate
- 物体初始位置来源
  - `replay_m_obj_pose_debug/d_pour_blue_1/multi_object_world_poses.npz`
  - 对应关键帧 1 的 `cup__pose_world_*`、`bottle__pose_world_*`
- action 阶段物体运动来源
  - 由于你设置了 `--replay_objects_during_action 1`
  - 所以后续物体不会绑定在 TCP 上
  - 而是继续按 `multi_object_world_poses.npz` 里的世界轨迹 replay

1. 先看 `plan_summary.json`
   - 确认最终执行的 `selected_candidates[*].pose_world_wxyz`
   - 确认是不是你手工指定的 candidate
2. 再看 `selected_candidates[*].raw_pose_world_wxyz`
   - 判断问题出在原始相机系转世界系，还是出在 `candidate_target_local_x_offset_m`
3. 再核对 `multi_object_world_poses.npz`
   - 因为你的物体 action 阶段是 replay 世界轨迹，不是 TCP 附着
4. 如果要看执行时每一帧的真实世界系状态
   - 打开 `--save_pose_debug 1`
   - 重点看 `pose_debug.jsonl` 里的 `current_left_tcp_pose_world_wxyz`、`current_right_tcp_pose_world_wxyz`、`object_actor_poses`

对当前这条命令，最关键的一句可以概括成：

- 手臂走的是 “AnyGrasp 候选变到世界系之后的目标轨迹”
- 物体走的是 “replay 文件里的世界系物体轨迹”
- 手部文件只参与“候选排序”，不直接生成执行位姿
