## 4. Run AnyGrasp Keyframe Planner In RoboTwin

Purpose:
- Load AnyGrasp grasp JSONs.
- Match candidates against hand orientation.
- Replay selected grasp/action keyframes in RoboTwin.

Main entry:
- `code_painting/run_plan_anygrasp_keyframes_r1_batch.sh`

Feature:
- dual-arm synchronized execution
- `urdfik` backend
- optional manual candidate overrides
- stage replan-until-reached

Example:
```bash
CUDA_VISIBLE_DEVICES=3 bash /home/zaijia001/ssd/RoboTwin/code_painting/run_plan_anygrasp_keyframes_r1_batch.sh \
  /home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_batch_results \
  /home/zaijia001/ssd/RoboTwin/code_painting/replay_m_obj_pose_d_pour_blue \
  /home/zaijia001/ssd/data/R1/gt_depth_vis/d_pour_blue/hand_vis \
  /home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_plan_keyframes \
  --ids 1 \
  --keyframes 1 21 \
  --arm auto \
  --execute_both_arms 1 \
  --planner_backend urdfik \
  --manual_candidate 1 left 5 \
  --manual_candidate 1 right 11 \
  --replan_until_reached 1 \
  --replan_until_reached_max_attempts 20 \
  --reach_error_pose_source ee \
  --init_prefix_frames 15 \
  --pause_after_keyframe1_seconds 5 \
  --enable_viewer 1
```

Key parameters:
- `--keyframes a b`: keyframe-1 grasp target and keyframe-2 action target
- `--execute_both_arms 1`: synchronize both arms; both must finish each stage before the next stage
- `--planner_backend urdfik|curobo`: Backend A direct IK or Backend B full cuRobo MotionGen
- `--urdfik_trajectory_mode joint_interp|cartesian_interp_ik`: within Backend A, choose the original "single endpoint IK + joint interpolation" mode or the new "Cartesian TCP interpolation + waypoint IK" mode
- `--urdfik_cartesian_interp_steps N`: number of TCP Cartesian waypoints used by `--urdfik_trajectory_mode cartesian_interp_ik`
- `--manual_candidate FRAME ARM IDX`: manually override one candidate
- `--reach_error_pose_source ee|tcp`: choose wrist/endlink or fingertip TCP for error computation
- `--replan_until_reached 1`: keep replanning from current state until tolerance or attempt limit
- `--init_prefix_frames N`: prepend fixed init frames before motion starts
- `--pause_after_keyframe1_seconds 5`: after keyframe-1 is reached and the gripper is closed, hold that pose for 5 seconds before planning/executing keyframe-2
- `--replay_objects_during_action 1`: replay object tracks from `replay_dir/multi_object_world_poses.npz` during the `action` stage instead of attaching selected objects to the TCP
- `--replay_objects_ignore_collision 1`: create replayed objects as visual-only kinematic actors and ignore arm/object collisions
- `--object_mesh_override NAME=/abs/path/to/mesh.obj`: override the display mesh for one replayed object, e.g. replacing a broken `cup.obj` with `blue_cup.obj`
- `--candidate_keep_camera_up 1`: resolve only the upside-down ambiguity around the gripper forward axis and keep the gripper-mounted camera/top side on the upward side overall, without forcing one exact roll
- Current keyframe rule: keyframe 1 keeps the more upward-facing roll variant as the baseline; later keyframes only choose between `original / local +X 180-degree flip`, selecting the smaller change relative to the previous selected keyframe to avoid a full extra roll at keyframe 22
- `--candidate_camera_top_axis z|y`: choose which local axis should be treated as the gripper top axis for the camera-up rule; current default recommendation is `z`
- `--candidate_target_local_x_offset_m VALUE`: shift each AnyGrasp planning target along the gripper local `+X` axis before visualization and planning. Use `-0.12` if the raw candidate behaves like a wrist/endlink pose and you want the planner input to represent fingertip TCP instead

Notes:

- During `pregrasp / grasp / pause_after_keyframe1`, objects remain fixed at keyframe-1 poses so the first target does not drift away before grasping.
- When `--replay_objects_during_action 1` is enabled, objects are no longer attached to the TCP; they follow the recorded `multi_object_world_poses.npz` trajectory directly.
- To compare "camera used when exporting world poses" against "camera used by the planner now", first regenerate `multi_object_world_poses.npz` with the updated `render_multi_object_pose_r1_npz.py`; it now stores `head_camera_pose_world_wxyz`.
- On the planner side, add:
  - `--save_pose_debug 1`
  This writes `output_dir/pose_debug.jsonl` with:
  - current `head_camera_pose_world_wxyz`
  - current left/right TCP poses
  - current object actor poses
  - replay-export `head_camera_pose_world_wxyz` for the same active frame
- The planner now also writes `output_dir/debug_execution_metrics.jsonl` during execution-preview recording. This is a lighter-weight per-frame analysis log than `pose_debug.jsonl` and focuses on:
  - per-arm target pose vs current pose
  - planned object pose vs actual object pose
  - target-object and current-object xyz deltas and distances
- The planner now also writes PNG charts under `output_dir/analysis_plots/`:
  - `{left,right}_target_current_error_vs_time.png`
  - `{left,right}_object_distance_vs_time.png`
  These plots use time on the x-axis and are meant for the exact debugging case "how does target-vs-current arm error evolve over time?".
- `debug_execution_preview.mp4` now includes a colored metric panel:
  - yellow: stage/object header
  - cyan: target minus current pose
  - red: planned object minus actual object
  - orange: target minus object
  - green: current pose minus object
- If keyframe-1 and keyframe-2 differ mostly by a redundant `360`-degree roll around the gripper forward axis, enable:
  - `--candidate_keep_camera_up 1`
  - `--candidate_camera_top_axis z`
  This keeps the local forward axis fixed and resolves only the free roll so the gripper-mounted camera stays on top.

## 6. Generate Raw AnyGrasp Annotated Preview Without RoboTwin

Purpose:
- Do not open RoboTwin or SAPIEN.
- Inspect the candidate pool in three stages on top of AnyGrasp images or replay RGB frames.
- Stage 1 prints before/after left-right object-filter counts in the terminal.
- Stage 2 exports a combined left-right orientation-ranked image.
- Stage 3 exports a combined left-right fused-score-ranked image.

Main entry:
- `code_painting/render_anygrasp_ranked_preview.py`

Example:
```bash
/home/zaijia001/ssd/miniconda3/envs/RoboTwin_bw/bin/python \
  /home/zaijia001/ssd/RoboTwin/code_painting/render_anygrasp_ranked_preview.py \
  --anygrasp_dir /home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_batch_results/d_pour_blue_1 \
  --replay_dir /home/zaijia001/ssd/RoboTwin/code_painting/replay_m_obj_pose_d_pour_blue_norobot/d_pour_blue_1 \
  --hand_npz /home/zaijia001/ssd/data/R1/gt_depth_vis/d_pour_blue/hand_vis/hand_detections_1.npz \
  --base_image_dir /home/zaijia001/ssd/RoboTwin/code_painting/replay_m_obj_pose_d_pour_blue_norobot/d_pour_blue_1/head_anygrasp_frames \
  --base_image_mode raw \
  --output_dir /home/zaijia001/ssd/RoboTwin/code_painting/anygrasp_direct_preview/d_pour_blue_1 \
  --frames 1 21 \
  --top_k 2 \
  --left_target_object cup \
  --right_target_object bottle \
  --draw_grasp_boxes 1
```

Outputs:
- `frame_000001_left_right_orientation_rank.png`
- `frame_000001_left_right_fused_rank.png`
- `summary.json`

Notes:
- `--base_image_dir .../head_anygrasp_frames --base_image_mode raw` is the recommended mode when you want a clean image without AnyGrasp's original dense gripper overlay.
- For the current `d_pour_blue` preview workflow, prefer `replay_m_obj_pose_d_pour_blue_norobot`.
- The older non-`norobot` replay root can contain problematic exports and was one source of the earlier preview errors.
- `--replay_dir` is now optional:
  - with it, the script runs staged mode
  - without it, the script falls back to the original raw-score combined preview mode
- In staged mode, left/right object assignment does not use different distance formulas.
  - each candidate first gets one shared `nearest_object` from world-space candidate-to-object distances
  - left/right separation happens afterward through `left_target_object/right_target_object`
- The script prefers `head_camera_pose_world_wxyz` from `replay_dir/multi_object_world_poses.npz`.
- If an older replay export does not contain that field, it now falls back directly to a fixed `head_camera_pose_world_wxyz`, without launching the simulator, and writes both `camera_pose_world_wxyz` and `camera_pose_source` into `summary.json`.
- Replay object center markers and object-name labels are now controlled by `--draw_object_overlay 1`; they are off by default so staged mode matches score-only mode visually.
- `--debug_dump_object_distances 1` exports `frame_<frame>_object_distance_debug.json` for per-candidate world-space object-distance inspection.
- That debug JSON now also includes a top-level `object_world_positions` block for the current frame.
- The terminal debug stream now prints `camera_world`, frame-level `objects={...}`, and per-candidate `gripper_world=[x,y,z]`.
- Staged mode now logs object partitioning before hand assignment:
  - `object-partition`: shared candidate counts per nearest object, for example `cup=12 bottle=8`
  - `arm-map`: object-to-hand mapping result, for example `left<-cup:12 right<-bottle:8`
- Staged mode now also logs orientation filtering after hand mapping:
  - `orientation-filter`: for example `left_before=12 left_after=2 right_before=8 right_after=2 max_rot_deg=90.0`
- If the replay object poses are obviously off-scene, the script prints an `object-world-warning` line so you can tell the problem is in the replay export rather than a small grasp transform error.
- `frame_<frame>_object_distance_debug.json` now stores shared `object_candidates` instead of duplicated left/right candidate-distance lists.
- Left and right are no longer saved as separate files; the script writes combined top/bottom images only.
- `--draw_grasp_boxes 1` draws a lightweight gripper wireframe for each displayed candidate on top of the raw RGB image.
- `--draw_hand_reference 1` is enabled by default and draws the human-hand reference gripper pose in a different color for each arm.
- The panel header now shows `hand ref idx=<frame>` so you can tell which valid hand frame was actually used for orientation comparison.
- The right-side panel now shows formulas and substituted arithmetic directly instead of compact `ori/rot` labels.
- `--max_rotation_distance_deg` defaults to `90.0`; set it below `0` if you want to disable this orientation filter.
- The default fused score is:
  - `0.5 * anygrasp_score + 0.5 * orientation_score`

## 7. Combined Object + Gripper Replay Logic

Reference document:
- `agent-read/V1.12_object_and_gripper_replay_logic.md`

Summary:
- During `pregrasp/grasp/pause`, objects stay at keyframe-1 world pose.
- During `action`, if `--replay_objects_during_action 1` is enabled, objects are replayed from `multi_object_world_poses.npz` in world coordinates.
- In that mode, objects no longer follow the TCP.
